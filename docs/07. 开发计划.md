# 开发计划

本文档定义 SmartFlow Flutter 重写阶段的开发路线。目标是在不照搬原 Web 项目接口设计的前提下，复用其已验证的业务逻辑和场景经验，逐步建立适合移动端、Flutter、drift 与 Riverpod 的实现结构。

## 1. 开发原则

### 1.1 领域优先，垂直切片验证

SmartFlow 的核心复杂度在复式记账、交易场景、余额一致性、统计口径和更正链路，因此开发应从领域模型和账务过账能力开始。

但不采用“先把全部底层能力一次性做完，再接 UI”的方式。每完成一组核心领域能力，应通过一个薄 UI 流程验证：

- drift 事务边界是否合理
- Riverpod 状态粒度是否合适
- 页面返回、刷新、错误展示是否顺畅
- 领域接口是否贴合移动端真实交互
- 查询模型是否能支撑列表、详情和统计展示

### 1.2 接口从业务用例长出来

原 Web 项目只作为业务行为样本，不作为接口设计模板。

底层接口应从以下来源推导：

- 《03.2 核心业务规则》定义的业务动作
- 《03.3 业务场景流程示例》定义的入账结果
- 每个业务动作必须维护的数据不变量
- Flutter 页面真实需要发起的命令与读取的视图
- 测试编写是否自然

不提前设计完整接口集。新增接口必须有明确用例支撑。

### 1.3 写入与查询分离

写入接口负责维护业务不变量和账务一致性；查询接口负责提供 UI 读模型。页面层不直接拼装 `entries`，也不直接绕过领域服务写数据库。

```text
页面 / provider
  -> command service / query service
  -> repository interface
  -> drift implementation
```

### 1.4 先稳定核心闭环，再扩展复杂能力

优先完成“账户 + 分类 + 普通交易 + 流水 + 余额”的最小闭环。复杂业务、预算、导入、自动化能力在核心闭环稳定后再进入。

## 2. 阶段路线

## 阶段 0：工程骨架与基础设施

目标：建立可持续开发的 Flutter 工程骨架。

范围：

- Flutter Android 工程初始化
- Material 3 主题入口与设计系统目录
- Riverpod、go_router、drift、freezed、json_serializable 等基础依赖
- drift 数据库打开、迁移入口和测试数据库能力
- 顶层 provider 装配
- 基础错误类型、Result / Failure 或等价错误处理策略
- Money 类型、金额格式化、月份键、日期工具

验收：

- 应用可启动
- 可运行单元测试
- drift in-memory 测试可用
- 目录结构符合《06. 项目结构说明》

## 阶段 1：账务核心模型

目标：实现复式记账最小底座。

范围：

- 领域实体与枚举：
  - Account
  - Transaction
  - TransactionDetail
  - Entry
  - Budget
  - AccountType
  - EntryDirection
  - BusinessPurpose
  - MutationKind
  - BusinessState
- drift 表：
  - accounts
  - transactions
  - transaction_details
  - entries
  - budgets
- 借贷方向规则
- 分录平衡校验
- 账户余额 delta 计算
- 统一过账能力：
  - 写入 transaction
  - 写入 transaction_details
  - 写入 entries
  - 校验借贷平衡
  - 同事务更新账户余额

建议接口：

```dart
abstract interface class PostingService {
  Future<PostTransactionResult> post(PostTransactionCommand command);
}
```

验收：

- 普通支出、普通收入、普通转账可通过领域测试完成入账
- 分录不平衡时拒绝写入
- 账户余额与 entries 推导结果一致
- 写账过程在同一数据库事务内完成

## 阶段 2：最小业务闭环

目标：完成第一个可运行的移动端记账闭环。

范围：

- AccountService
  - 创建账户
  - 编辑账户基础信息
  - 期初余额
  - 账户列表
- CategoryService
  - 创建收入 / 支出分类
  - 分类树查询
- TransactionService
  - 普通支出
  - 普通收入
  - 普通转账
- TransactionQueryService
  - 流水列表
  - 交易详情
  - 账户流水
- 最小 UI：
  - 首页或账户概览
  - 账户管理
  - 分类管理
  - 新增交易
  - 流水列表
  - 交易详情

验收：

- 用户可以创建账户和分类
- 用户可以记录支出、收入、转账
- 账户余额实时更新
- 流水列表按时间倒序展示
- 交易详情可查看业务信息和分录信息
- 页面刷新和返回后数据状态正确

## 阶段 3：复杂交易场景

目标：补齐原项目已验证的核心复合交易能力。

范围：

- 退款
- 报销垫付
- 报销到账
- 结束报销
- 还款：
  - 本金
  - 利息
  - 手续费
- 借入
- 余额调整

需要保留的业务不变量：

- 退款总额不得超过当前有效原支出金额
- 报销到账不得超过待核销余额
- 结束报销后该报销链路不可继续普通到账
- 报销差额按《03.2 核心业务规则》入账
- 还款本金影响负债账户，利息和手续费影响支出分类
- 借入同时增加资产和负债；无到账资产的债务期初使用系统期初余额账户对冲

验收：

- 《03.3 业务场景流程示例》中对应场景均有测试覆盖
- 复合交易在流水列表中可读
- 子记录默认可折叠到父交易详情
- 账户余额、收支统计、预算统计口径不互相污染

## 阶段 4：更正、删除与历史追溯

目标：实现已入账交易不可直接覆盖的审计链路。

范围：

- 更正 = 冲销原交易 + 重记正确交易
- 删除 = 冲销原交易且不补记
- 红字冲销
- 更正链查询
- 默认流水隐藏历史记录和冲销记录
- 详情页展示更正、冲销与关联子记录链路

需要保留的业务不变量：

- entries 写入后不直接 update / delete
- 已入账交易的金额、账户、分类变化必须通过更正链表达
- 删除不物理删除已入账记录
- 存在子记录的父交易更正必须保护子记录依赖的数据关系
- 删除父交易时，其当前有效子记录也必须一致性处理

验收：

- 多次更正链路可追溯
- 删除后账户余额被冲销抵消
- 默认列表只展示当前有效业务记录
- 详情页可查看完整历史链路

## 阶段 5：预算与统计

目标：在核心交易稳定后建立统计和预算能力。

范围：

- 总预算
- 分类预算
- 预算进度
- 预算趋势
- 收入 / 支出月度统计
- 分类占比
- 同环比基础指标
- `is_excluded_from_stats`
- `is_excluded_from_budget`

验收：

- 统计只从 entries 推导
- 退款自然抵减支出
- 报销垫付不计入个人支出
- 报销差额按收入或支出口径进入统计
- 预算排除按整笔交易粒度生效
- 分类迁移后预算归属和历史统计可追溯

## 阶段 6：移动端体验完善

目标：将已验证的业务能力打磨为稳定移动端体验。

范围：

- 移动端表单体验
- 金额输入体验
- 日期时间选择
- 分类与账户选择器
- 筛选与搜索
- 空状态、错误状态、加载状态
- 暗色模式
- 无障碍与触控目标
- 常用业务组件沉淀

验收：

- 核心记账路径操作流畅
- 高频入口层级足够短
- 错误提示可理解
- 金额、正负号、财务语义色表达一致
- 组件复用符合《05. 设计系统》

## 阶段 7：数据管理与导入

目标：在核心账务模型稳定后接入导入与备份能力。

当前阶段暂不实现支付宝、微信、银行导入与智能分类，仅保留后续扩展空间。

优先范围：

- 本地数据库备份
- 本地数据库恢复
- CSV / Excel 导出
- 一木记账导入
- 导入预览
- 字段映射
- 提交前调整

后续可按需要补充：

- 导入批次记录
- 来源行追踪
- 映射记忆
- 去重策略
- 更多账单来源模板

验收：

- 导入结果遵循同一交易写入规则
- 导入交易与手动交易共享更正、删除、统计和预算规则
- 导入失败不会产生半写入数据
- 用户可在提交前理解将生成哪些交易

## 3. 暂缓范围

以下内容不作为当前核心开发目标：

- 周期性账单
- 用户自定义规则引擎
- AI 辅助分类
- 支付宝 / 微信 / 银行账单完整模板库
- WebDAV 或云同步
- 多账本
- 家庭账本与多人协作
- 投资行情同步与收益率计算
- iOS 支持

这些能力可以在核心账务模型稳定后，以独立 feature 形式逐步引入。

## 4. 接口演进策略

### 4.1 命令接口

命令接口表达系统允许用户发起的业务动作。

示例：

```dart
abstract interface class TransactionCommandService {
  Future<PostTransactionResult> createExpense(CreateExpenseCommand command);
  Future<PostTransactionResult> createIncome(CreateIncomeCommand command);
  Future<PostTransactionResult> createTransfer(CreateTransferCommand command);
}
```

原则：

- command 字段来自业务动作，不来自页面控件
- command 不暴露 drift、DAO、SQL、表结构细节
- command service 负责校验业务不变量
- 新业务场景出现后再新增命令接口

### 4.2 查询接口

查询接口服务 UI 读模型。

示例：

```dart
abstract interface class TransactionQueryService {
  Future<List<TransactionListItem>> listTransactions(TransactionListQuery query);
  Future<TransactionDetailView> getTransactionDetail(int transactionId);
}
```

原则：

- 查询模型可以面向页面展示
- 查询接口不承担写账逻辑
- 页面不直接读取 entries 后自行解释业务语义
- 复杂统计优先沉淀到 domain/stats 或 query service

### 4.3 Repository 接口

Repository 是领域层到持久化层的端口，不是页面 CRUD 集合。

原则：

- 从 service 的真实需要中增加方法
- 不提前铺满所有表的 CRUD
- 事务能力必须显式提供
- drift 实现细节不泄漏到 domain

## 5. 测试策略

测试优先级：

1. 领域规则测试
2. drift repository 测试
3. Riverpod provider 测试
4. Widget 测试

核心测试来源：

- 《03.3 业务场景流程示例》
- 原 Web 项目已验证的业务场景
- 更正、删除、退款、报销、还款等高风险路径
- 金额边界、币种不一致、账户类型不合法等错误路径

每个阶段至少满足：

- 核心 happy path 有测试
- 关键不变量有测试
- 已发现的业务缺陷补回归测试

## 6. 当前推荐起点

当前最适合从以下闭环开始：

```text
Money / enums / drift schema
  -> PostingService
  -> AccountService + CategoryService
  -> createExpense / createIncome / createTransfer
  -> TransactionQueryService.listTransactions
  -> 最小账户、分类、记账、流水 UI
```

这个闭环完成后，再进入退款、报销、还款和更正删除。这样既能复用原项目的业务经验，又能尽早验证 Flutter 移动端架构是否自然。
