# 08.2 通用 UI 与业务域接入协议

## 适用范围

本决策约束**通用 UI**（如交易详情页、列表页）与**业务域**（08.1 定义；分期、订阅、保险等独立业务域）之间的接入方式。

不约束：

- UI 与账务核心 / 流图原语的关系（按 `BusinessPurpose` switch 渲染照常）；
- 业务域各自内部的 UI（如分期合同详情页内的渲染逻辑）。

## 背景

08.1 已定 UI → 业务域 → 账务核心 的单向依赖关系，但还需要明确通用 UI 如何判断"这笔交易归哪个业务域管"，以及删除、编辑、改账户、改时间、改备注等动作该走哪条服务路径。

早期方案使用 `TransactionHandlers + OrchestrationContribution + OrchestrationRegistry`：

- 每个业务域提供 contribution，按 `transactionId` 反查是否命中；
- registry 逐个询问 contribution；
- UI 拿到 handler 后执行 universal 动作。

该方案能解耦 UI 与具体业务域，但问题是：业务归属依赖异步反查，而不是交易自身携带的稳定元数据；同时 contribution / registry 只是"按交易归属选 policy"的旧名字，和最终抽象不一致。

当前决策改为：交易表保存开放式归属元数据，通用 UI 直接按 `transaction.owner_type` 解析 `TransactionActionPolicy`。

## 决策

通用 UI 通过 **`TransactionActionPolicy` + `transactions.owner_*`** 与业务域解耦。

> **TransactionActionPolicy 是集成层契约**——它的存在前提是"有这个 UI"，离开交易详情页则毫无意义。它把每个业务域的领域能力组合成"交易详情页可执行的统一动作集"，本质是 UI 接入业务域的方式，不属于任何业务域。整套（接口、各业务域实现、provider）位于 `features/transactions/action_policy/` 与 `features/transactions/providers/`。

### 归属元数据

`transactions` 保存三列业务归属字段：

- `owner_type`：业务域类型，如 `installment`；普通交易为 `null`。
- `owner_id`：业务主体 id，如 `installment_contracts.id`。
- `owner_role`：该交易在主体内的角色，如分期的 `disbursement / regular_repayment / extra_principal / early_settlement`。

`BusinessPurpose` 继续表示账务行为；`owner_*` 只表示业务归属。`source_kind` 继续表示创建渠道，不承担业务归属语义。

账务核心只保存和透传 `owner_*`，不解释具体 `owner_type / owner_role`，不根据它们改变过账规则。

### Action Policy

`TransactionActionPolicy` 定义所有交易都拥有的 universal 动作槽位：

- `delete()` —— 删除当前交易；
- `editRoutePath()` —— 编辑入口的路由路径；
- `changeSettlementAccount(accountId)` —— 改 paidFrom / receive 账户；
- `changeOccurredAt(time)` —— 改交易时间；
- `changeNote(note)` —— 改备注；
- `canEdit(field)` —— 字段级可用性（返回带 reason 的 `EditPermission`）；
- `displayBanner()` —— 可选横幅文案。

Provider 解析规则：

- `owner_type == null` → `DefaultTransactionActionPolicy`；
- `owner_type == 'installment'` → `InstallmentTransactionActionPolicy`；
- 已有 owner 但未注册策略 → `UnknownOwnedTransactionActionPolicy`。

`UnknownOwnedTransactionActionPolicy` 只允许改备注；删除、编辑跳转、金额、账户、时间等可能破坏账务或未知业务状态的动作全部禁用，并展示"当前版本未识别业务来源"的 banner。

### 代码位置

整套 `TransactionActionPolicy` 体系位于 `features/transactions/` 集成层：

- `features/transactions/action_policy/transaction_action_policy.dart`：接口。
- `features/transactions/action_policy/default_transaction_action_policy.dart`：`DefaultTransactionActionPolicy`（普通交易）与 `UnknownOwnedTransactionActionPolicy`（兜底）。
- `features/transactions/action_policy/<域>_transaction_action_policy.dart`：各业务域实现，例如 `installment_transaction_action_policy.dart`。
- `features/transactions/providers/transaction_action_policy_provider.dart`：`transactionActionPolicyProvider`，按 `owner_type` 装配 policy。

### 依赖方向与边界

- features/transactions/ 可以 import 任何业务域的 service（features 是消费层，允许跨业务域消费）。
- domain/ 不感知 TransactionActionPolicy 的存在；将来业务域提炼为独立 pub 包时，policy 不跟随业务域走，因为它依赖本 app 的 UI 形态。
- policy 实现 import 自己服务的业务域 service（如 InstallmentTransactionActionPolicy 只 import InstallmentService），不持有 `TransactionService` —— 业务一致性逻辑必须在业务域 service 内完成。

详细 import 边界与跨业务域约束见《08.1 业务域划分与账务核心独立性》。

### 动作的二分

| 类型 | 决定可见性 | 决定具体行为 | 例子 |
| :--- | :--- | :--- | :--- |
| **purpose-bound** | `BusinessPurpose`（UI 按 switch 渲染） | 流图原语 service | 退款（dailyExpense / reimbursementAdvance）、报销结案（reimbursementAdvance） |
| **universal** | 永远可见 | `TransactionActionPolicy` 提供 | 删除、编辑跳转、改账户、改时间、改备注 |

**purpose-bound 动作不进 `TransactionActionPolicy`**。它们由通用 UI 按 `BusinessPurpose` 自行渲染，跟业务域无关。

### 调用方向

```text
通用 UI ─→ TransactionActionPolicy (interface)
              ▲
              │  selected by transaction.owner_type
              │
   ┌──────────┴────────────────────┐
   │                               │
DefaultTransactionActionPolicy   InstallmentTransactionActionPolicy ─→ 分期域 service
(走 transactionService)
```

通用 UI（`transaction_detail_page` 等页面 widget）只依赖 `TransactionActionPolicy` 接口。具体 policy 实现的 import 与装配集中在 `features/transactions/providers/transaction_action_policy_provider.dart`；详情页 widget 不直接 import 任何 `<域>TransactionActionPolicy`。

业务域专属 policy（如 `InstallmentTransactionActionPolicy`）**只持有自己所服务业务域的 service**，不持有 `TransactionService`。"先调核心 service 再调业务域 service" 这种胶水编排属于业务一致性逻辑，必须下沉到对应业务域的 service 内部。policy 的职责退化为两件事：
1. 按 `ownerRole` 把通用动作路由到业务域 service 的领域入口；
2. 提供策略声明（`canEdit` / `displayBanner` / `editRoutePath`）。

### 分期 owner 约定

分期交易使用：

- `owner_type = 'installment'`
- `owner_id = installment_contracts.id`
- `owner_role = disbursement / regular_repayment / extra_principal / early_settlement`

分期 policy 的关键行为（按 `ownerRole` 路由到 `InstallmentService` 的领域入口；policy 自身不持有 `TransactionService`，所有写入都收口到分期域 service 内部）：

- 放款交易 `delete()` → `InstallmentService.deleteContract`；
- 还款交易 `delete()` → `InstallmentService.revertRepayment`；
- 放款交易改账户 / 改时间 / 改备注 → `InstallmentService.updateContract`（service 内部按字段决定是否重算 schedule，并联动放款交易的 entry / occurredAt / note）；
- 还款交易改账户 / 改时间 / 改备注 → `InstallmentService.editRepayment`（service 内部委托 `TransactionService` 完成 transactions 表写入，未来若需联动合同状态在此处加）；
- 金额修改 → 禁止，引导到合同详情。

## 反例

### 反例一：通用 UI 维护 `if (installmentLink != null)` 分支

- **形式**：`transaction_detail_page` 持有 `installmentLink / subscriptionLink` 等状态，按 link 类型嵌套 if/else 替换 UI 组件。
- **为什么不取**：每加一个业务域都要改通用 UI；通用 UI 被迫识别每个业务域身份；跨业务域的 UI 一致性容易漂移。

### 反例二：继续保留 `OrchestrationContribution / OrchestrationRegistry`

- **形式**：进入详情页时逐个 contribution 反查 `transactionId` 是否命中，再返回 handler / policy。
- **为什么不取**：归属已经是 `transactions.owner_*` 的一等字段，registry 反查会形成第二套归属判断机制；命名仍停留在"编排装配"而不是"动作策略"；也不利于未知 owner 的统一兜底。

### 反例三：把 purpose-bound 动作塞进 `TransactionActionPolicy`

- **形式**：policy 接口里加 `refund()` / `closeReimbursement()` 之类的方法，每个业务域都给 no-op。
- **为什么不取**：混淆 universal 与 purpose-bound；接口随 purpose 数量膨胀；业务域被迫实现不属于自己的方法。

### 反例四：用 redirect 替代就地操作

- **形式**：受管交易的 ActionBar 只放"去 X 页"跳转按钮，用户被踢出当前页面才能完成删除 / 编辑等操作。
- **为什么不取**：同类交易在不同上下文行为不一致；用户无法在原位完成原本就该原位完成的操作；跳转可以是 policy 提供的编辑路径，但不应替代 universal 动作。

### 反例五：policy 跨多个业务域 service 自行编排一致性

- **形式**：`XxxTransactionActionPolicy` 构造时同时注入对应业务域 service 与 `TransactionService`（或其它业务域 service 作为 fallback）；在 `changeSettlementAccount` 之类的 universal 动作里，policy 先调 `transactionService.updateTransactionBasics` 写 entry，再调业务域 service 同步业务主体字段。
- **为什么不取**：
  - 业务一致性逻辑（先改交易再同步主体 / 改某字段同时联动子表重算）被埋在集成层 policy 里，调用顺序、失败处理、未来扩展都散落在 policy 的各 universal 方法中；
  - policy 同时认识两层抽象（账务核心 service 与业务域 service），变成胶水层；业务域 service 接口演进会被迫同步改 policy；
  - 同一份联动逻辑无法被该业务域的其他入口（合同详情页、还款表单等）复用——那些入口不可能走 policy。
- **替代方案**：业务域 service 对外提供领域语义的编辑入口（如分期的 `updateContract` / `editRepayment`），内部完成"主体字段 + 关联交易 + 计划表"的一致性维护；policy 退化为"按 `ownerRole` 路由到这些入口" + 策略声明，只 import 自己服务的业务域 service，不持有 `TransactionService`。

## 归类判断

新动作 / 新功能接入 UI 时，按以下次第判断归属：

1. **它是否依赖某个具体 `BusinessPurpose`？**
   - 是 → **purpose-bound**，进通用 UI 的 purpose switch，跟 policy 无关。
2. **它是否所有交易都有 / 都可能有？**
   - 是 → **universal**，进 `TransactionActionPolicy` 接口；各 owner 的 policy 按需重写，默认实现走 `TransactionService`。
3. **它是否只属于某个业务域的主体操作？**
   - 是 → 不进通用 UI，归业务域自己的 feature 页面。

任何让交易详情页 widget 直接 `import` 具体 `<域>TransactionActionPolicy` / 识别具体 link 子类型的方案 → 拒绝。
任何让 `TransactionActionPolicy` 接口为 purpose-bound 动作开不可选槽位的方案 → 拒绝。
任何重新引入 `OrchestrationContribution / OrchestrationRegistry` 作为交易归属判断入口的方案 → 拒绝。
任何让 `<域>TransactionActionPolicy` 持有 `TransactionService` 或其它业务域 service 并自行编排跨 service 一致性的方案 → 拒绝。
任何把 `TransactionActionPolicy` 接口或具体实现放回 `domain/` 的方案 → 拒绝（它是集成层契约，不属于任何业务域）。

## 演进规则

- **加新 universal 动作**：扩 `TransactionActionPolicy` 接口 + 所有现有实现跟进。
- **加新业务域**：按《08.1》在 `domain/` 与 `data/` 创建独立业务域目录；定义 `owner_type / owner_role` 约定；在 `features/transactions/action_policy/` 新增 `<域>_transaction_action_policy.dart` 实现；在 `features/transactions/providers/transaction_action_policy_provider.dart` 注册 owner_type 到 policy 的映射。业务域 service 自身需要对外提供该域的领域编辑入口（合同 / 还款 / 订阅 / 凭证…），policy 只做路由，不持 `TransactionService`。
- **某 purpose-bound 动作未来需要被业务域覆盖**：`TransactionActionPolicy` 增加一个可选 slot（默认 null = 走原 purpose-bound 渲染），policy 选择性重写。
- **未知 owner**：统一落到 `UnknownOwnedTransactionActionPolicy`，只允许低风险元数据操作。
- **跨业务域冲突**：当前假设一笔交易只有一个 owner；若未来真有交叉需求，再扩展 owner 模型，不预先设计。
