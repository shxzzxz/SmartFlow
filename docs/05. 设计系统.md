# 设计系统

本文档定义 SmartFlow Flutter 工程内的设计系统落地规则。约束视觉值、主题、组件和业务视觉语义在代码中的归属。

## 1. 当前阶段

SmartFlow 当前处于 MVP 阶段，采用 **A 级：最小可落地版**：

- 建立 token、theme、基础组件和业务组件的分层入口。
- 新增代码必须使用设计系统已有能力，不再扩散原始色值和魔法数。
- 存量代码不做一次性重构，采用“触达即治理”。

后续进入多人协作、组件数量明显增加或统计图表成为主要迭代区域时，升级到 **B 级：标准协作版**，补充组件示例页、静态检查规则和更稳定的页面模式库。

## 2. 分层结构

```text
Token 源层 -> 主题映射层 -> 基础组件层 -> 业务组件层
```

```text
lib/design_system/
├── tokens/
│   ├── colors.dart        # 原始色板，不被页面直接消费
│   ├── spacing.dart       # 间距 token
│   ├── radius.dart        # 圆角 token
│   ├── typography.dart    # 字体 token
│   ├── elevation.dart     # 阴影 token
│   ├── motion.dart        # 动效 token
│   └── chart.dart         # 图表色 token
├── theme/
│   ├── app_color_scheme.dart       # Material 3 ColorScheme
│   ├── app_theme_extension.dart    # 财务、状态、图表等扩展语义
│   ├── app_text_theme.dart         # TextTheme
│   └── app_theme.dart              # ThemeData 组装
├── widgets/              # 通用基础组件
└── showcase/             # 组件示例页，可选

lib/widgets/business/     # 跨 feature 复用的业务组件

assets/icons/             # 业务图标资源
├── account/
├── category/
└── tab/
```

### Token 源层

Token 源层是项目内视觉原始值的唯一来源，负责颜色、间距、圆角、字号、阴影、动效和图表色等基础值。

约束：

- token 命名按尺度或用途归类，如 `space4`、`radiusMd`、`durationFast`。
- 原始颜色只允许出现在 token 源层，业务页面禁止新增 `Color(0xFF...)`、hex、rgb 等写法。
- token 只定义值，不包含业务判断。

### 主题映射层

主题映射层将 token 映射为 Flutter 可消费的语义入口：

- Material 3 `ColorScheme`：品牌色、背景色、表面色、错误色等通用语义。
- `TextTheme`：标题、正文、标签等文本语义。
- `ThemeExtension`：Material 3 没有覆盖的状态、财务和图表语义。

消费边界：

- 颜色、文本、财务语义色优先通过 `Theme.of(context)`、`ColorScheme`、`ThemeExtension` 使用。
- 间距、圆角、动效时长等基础布局值允许通过 `AppSpacing`、`AppRadius`、`AppMotion` 直接使用。
- 页面层不直接读取原始色板 token。

### 基础组件层

基础组件放在 `lib/design_system/widgets/`，只表达通用 UI 能力，不包含财务业务含义。

新增基础组件前先扫描 `lib/design_system/widgets/` 下已有实现，避免重复造轮子

允许：

- `variant`：primary / outline / text 等。
- `size`：small / medium / large 等。
- `tone`：info / success / warning / danger 等通用状态。
- `state`：loading / disabled / selected 等。

禁止：

- 禁止出现 `IncomeButton`、`ExpenseBadge`、`AssetCard` 这类业务命名。
- 禁止在基础组件内部按账户类型、交易用途等业务字段改样式。
- 禁止在基础组件内部重新定义视觉 token。

### 业务组件层

业务组件放在 `lib/features/<feature>/widgets/` 或 `lib/widgets/business/`，用于沉淀财务领域的稳定表达。

典型组件：

- `MoneyText`：金额展示，统一正负号、颜色和格式。
- `MoneyInput`：金额输入，统一分位、小数、负数和校验反馈。
- `TransactionPurposeBadge`：交易用途标识。
- `TransactionBusinessStateBadge`：交易业务状态标识。
- `AccountTypeTag`：账户类型标签。

同类表达在 2 个以上页面重复出现时，应优先抽为业务组件。

## 3. 语义体系

### Material 语义

品牌、背景、表面、边框、禁用、错误等通用视觉语义优先使用 Material 3 `ColorScheme` 和 `TextTheme`，不在页面层另起命名。

### 状态语义

通过 `ThemeExtension` 提供：

- `success`
- `warning`
- `danger`
- `info`

`danger` 用于删除、失败、错误等通用危险状态，不等同于支出。

### 财务语义

通过 `ThemeExtension` 提供：

- `income`
- `expense`
- `transfer`
- `asset`
- `liability`
- `equity`

约束：

- `income` 不等同于 `success`。
- `expense` 不等同于 `danger`。
- `transfer` 保持中性表达，不在不同页面随意换色。
- `asset`、`liability`、`equity` 是账户域语义，不退化为具体色名。

### 图表语义

- 通用分析图使用 `chart1` ~ `chart5`。
- 明确表达财务概念的图表使用 `chartIncome`、`chartExpense`、`chartAsset`、`chartLiability`。
- 同一图表模块内不要混用系列语义和领域语义。

## 4. 编码规则

新增视觉值只能来自：

1. token 源层定义的常量。
2. 主题映射层暴露的 `ColorScheme`、`TextTheme`、`ThemeExtension`。
3. 基础组件或业务组件暴露的变体。

明确禁止：

- 在 widget 中新增长期存在的原始色值。
- 在页面层临时定义财务颜色、状态颜色或图表色。
- 在基础组件中写入财务业务判断。
- 在业务组件或页面中重新定义底层 token。

允许的临时例外：

- 历史 widget 中尚未迁移的原始样式。
- 第三方组件必须覆盖样式时，应优先封装统一适配层。
- 修复业务问题时，未触及区域可以暂不顺手重构。

原则：可以保留存量，不可以新增同类债务。

## 5. 平台与无障碍

- 当前阶段仅面向 Android，以 Material 3 为基础进行主题与组件扩展，不引入 Cupertino 风格组件。
- 支持跟随系统暗黑模式，亮色和暗色主题共用同一套 token 源。
- 财务正负表达不能只依赖颜色，应结合 `+` / `-`、文案或结构。
- 状态表达不能只依赖颜色，应辅以图标、文案或布局差异。

## 6. 演进规则

- 新增页面优先复用已有基础组件和业务组件。
- 新增财务视觉语义前，先补充本规范和 `ThemeExtension`。
- 高频重复模式先进入 `lib/features/<feature>/widgets/`，跨 feature 复用后迁移到 `lib/widgets/business/`。
- B 级阶段补充 `lib/design_system/showcase/`、`custom_lint` 或静态扫描，限制原始色值和 token 越层消费。
