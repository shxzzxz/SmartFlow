# 设计系统

## 文档定位

本文档是 SmartFlow 在 Flutter 工程内执行的项目级设计系统规范。

它回答的不是"颜色怎么选"，而是以下问题：

- 当前项目处于什么落地等级
- 视觉规则在 Flutter 代码中放在哪里
- 基础组件和业务组件如何分层
- 什么样的样式写法允许新增
- 视觉债务出现时如何治理

通用方法论（跨技术栈适用的设计系统落地原则）见《07. 通用设计系统落地规范》（待写）。本文档基于通用方法论收敛到 Flutter + Material 3 + Android 单端的具体落地。

## 当前项目落地等级

截至 2026-05-06，SmartFlow 处于 MVP 阶段，采用：

- **A 级：最小可落地版** 作为当前强制标准
- **B 级：标准协作版** 作为后续阶段升级目标

理由：

- 项目仍以单产品快速迭代为主
- Flutter + Material 3 + 自定义 ThemeExtension 已具备 token 化基础能力
- 业务组件还未沉淀，视觉一致性靠工程纪律保证

两条原则：

- 新增代码必须按设计系统执行，不再扩散原始色值与魔法数
- 旧代码不做一次性大重构，采用"触达即治理"的迁移策略

## 核心目标

- **视觉值来源唯一**：颜色、圆角、间距、字号、阴影、动效曲线集中在 token 源
- **财务语义稳定**：收入、支出、转账、资产、负债、权益等业务语义不在页面层临时发明
- **基础组件 / 业务组件分层**：避免业务含义写进通用组件
- **布局保持灵活，视觉语义保持收口**
- **文档可指导代码修改**，而非只描述理想状态

## 四层架构

```
Token 源层 → 主题映射层 → 基础组件层 → 业务组件层
```

### 第 1 层：Token 源层

承载位置：`lib/design_system/tokens/`

承载方式：以 Dart 常量与 `Color`、`TextStyle`、`Duration` 等强类型对象表达。

```
lib/design_system/tokens/
├── colors.dart        # 原始色板（不直接被业务消费）
├── radius.dart        # 圆角 token
├── spacing.dart       # 间距 token
├── typography.dart    # 字体 token
├── elevation.dart     # 阴影 token
├── motion.dart        # 动效 token
└── chart.dart         # 图表色 token
```

要求：

- 这是项目内**视觉原始值的唯一来源**
- 业务代码、widget、页面禁止再出现 hex / rgb / `Color(0xFFxxxxxx)` 形式的原始色值
- 字号、字重、行高、圆角值同样集中
- token 命名按尺度阶梯（如 `space2`、`space4`、`radiusMd`），不要带业务含义

### 第 2 层：主题映射层

承载位置：`lib/design_system/theme/`

```
lib/design_system/theme/
├── app_color_scheme.dart       # ColorScheme（Material 3 语义色）
├── app_theme_extension.dart    # 自定义 ThemeExtension：财务语义、图表语义
├── app_text_theme.dart         # TextTheme
└── app_theme.dart              # 组装 ThemeData
```

职责：

- 将 token 源映射为 Material 3 ColorScheme 的语义槽位（`primary` / `surface` / `onSurface` / `error` / ...）
- 把 token 源映射为自定义 `ThemeExtension`，承载 Material 3 没有的财务、图表语义
- `ThemeData` 由本层组装，应用层 `MaterialApp.theme` / `darkTheme` 直接消费

强制要求：

- 只做映射，不引入业务逻辑
- 优先使用语义名（`onSurface`、`incomeColor`、`assetColor`），不向业务暴露临时命名
- 图表色、状态色、财务业务色必须先在此层定义后才能使用

### 第 1 层与第 2 层的边界

- token 源**不直接被业务代码消费**
- 业务代码消费的是 `Theme.of(context).colorScheme.primary`、`Theme.of(context).extension<FinanceColors>()!.income`、`AppSpacing.md` 等语义值
- token 源中允许的"裸值"在主题映射层**包装一次**才能进入业务

### 第 3 层：基础组件层

承载位置：`lib/design_system/widgets/`

职责：

- 承载通用交互壳，基于 Material 3 widget 与变体表达通用 UI 语义
- 在 Material 3 默认行为基础上做轻量包装，提供项目统一的尺寸、形状、状态

允许内容：

- variant（如 `AppButton.primary` / `AppButton.outline` / `AppButton.text`）
- size（`small` / `medium` / `large`）
- tone（信息 / 成功 / 警告 / 危险）
- state（loading / disabled）
- layout（密度、对齐、边距）

禁止内容：

- 禁止把业务概念写进基础组件命名
- 禁止出现 `IncomeButton`、`ExpenseBadge`、`AssetCard` 这类命名
- 禁止在基础组件内部定义新的视觉 token

### 第 4 层：业务组件层

承载位置：`lib/features/<feature>/widgets/` 或 `lib/widgets/business/`

职责：

- 沉淀财务领域的复用表达
- 组合基础组件 + 财务语义

适合进入业务组件层的典型对象：

- `MoneyText`：金额展示，按符号、大小、颜色映射规则统一渲染
- `MoneyInput`：金额输入，统一处理小数、负数、千分位
- `TransactionPurposeBadge`：业务用途徽标
- `AccountTypeTag`：账户类型标签
- `TransactionLifecycleBadge`：生命周期状态标识

策略：

- 同一类表达在 2 个以上页面重复出现，应优先抽为业务组件而不是继续复制 widget
- 业务组件命名直接体现财务语义；基础组件保持中性

## 视觉约束 vs 布局约束

以下能力必须收口到 token、theme、组件变体：

- 颜色 / 背景色 / 边框色
- 圆角
- 阴影 / 高度
- 字体语义（字号、字重、行高）
- 图标尺寸默认值
- 动效曲线与时长
- 焦点态与按压态颜色

以下能力允许在页面层和业务层灵活使用：

- `Padding` / `EdgeInsets`
- `SizedBox` / `Spacer` / `Gap`
- `Row` / `Column` / `Wrap` / `Stack`
- `Flex` / `Expanded` / `Flexible`
- 响应式断点
- 对齐方式

判断原则：**视觉语义必须稳定，布局装配可以灵活。**

## 语义命名体系

### 1. 品牌语义（Material 3 ColorScheme）

直接使用 Material 3 内置语义：`primary` / `secondary` / `tertiary` / `surface` / `onSurface` / `outline` 等。

### 2. 状态语义

通过自定义 ThemeExtension 提供：

- `success`
- `warning`
- `danger`
- `info`

约束：

- `danger` 用于错误、删除、失败状态
- 不允许把 `expense` 直接当作 `danger` 的同义词

### 3. 财务业务语义

通过自定义 ThemeExtension 提供：

- `income`
- `expense`
- `transfer`
- `asset`
- `liability`
- `equity`

约束：

- `income` 表示财务收入表达，不等于通用成功态
- `expense` 表示财务支出表达，不等于危险操作
- `transfer` 必须保持中性，不允许在不同页面随意切换颜色
- `asset` / `liability` / `equity` 属于账户域语义，不应直接退化成 `green / red / blue`

### 4. 图表语义

混合策略：

- 通用分析图允许使用 `chart1` ~ `chart5`
- 明确表达财务概念的图表优先使用 `chartIncome` / `chartExpense` / `chartAsset` / `chartLiability`

禁止：

- 禁止页面临时写"这个图先随便用蓝绿红"
- 禁止同一图表模块一会用系列语义、一会用领域语义

## 新增代码的强制规则

### 允许的视觉来源

新增视觉值只能来自以下三类来源：

1. token 源层定义的常量
2. 主题映射层暴露的语义值（ColorScheme + ThemeExtension）
3. 基础 / 业务组件层暴露的变体

### 明确禁止

- 禁止在 widget 中新增 `Color(0xFF...)` 等原始色作为长期方案
- 禁止通过 `BoxDecoration` 直接写 hex 色值
- 禁止在基础组件中写入业务判断（如根据 `accountType` 改色）
- 禁止在业务组件或页面中重新定义底层 token
- 禁止跳过 ThemeExtension 直接读取 token 源

### 允许的临时例外

- 历史 widget 中尚未迁移的原始色板写法
- 第三方组件覆盖样式（如 fl_chart 的内部默认色），需在统一适配层包装
- 修复业务问题最小范围触达旧代码时，未触及区域可暂不顺手全改

例外原则：**可以保留存量，不可以新增同类债务。**

## 平台与无障碍

### 1. 平台范围

- 当前阶段仅 Android，UI 风格统一使用 Material 3
- 不引入 Cupertino 风格 widget
- iOS 加入支持后再讨论双风格策略

### 2. 暗黑模式

- 支持系统跟随暗黑模式
- 暗色主题与亮色主题共用同一套 token 源、不同的 ColorScheme 与 ThemeExtension 实例
- 财务业务语义（`income` / `expense` 等）在两套主题下均需保证可读对比度

### 3. 无障碍最低要求

- 状态表达不能只依赖颜色（必须辅以图标 / 文案 / 结构）
- 关键交互必须保留 `Focus` 焦点态
- 财务数据的正负表达，除颜色外应结合符号、文案或结构（如金额前缀 `+` / `-`）
- 触控目标最小尺寸遵循 Material 3 推荐（48dp）

## 工具链策略

### 当前阶段强制执行

- Code review 时检查是否新增原始色值
- Code review 时检查是否把业务语义写进基础组件
- 新增页面或新增组件优先消费已有基础 / 业务组件
- 业务层新增财务语义前，先补到本规范和 ThemeExtension 中

### 当前阶段推荐执行

- 启用 `flutter_lints` 默认规则集
- 视情况引入 `custom_lint` 限制 token 源外部直接消费
- 维护设计系统示例页（`lib/design_system/showcase/`），便于对照

## B 级升级触发条件

满足以下任一条件时，应升级到 B 级标准协作版：

- 团队人数达到 3 人及以上
- 基础组件数量明显增长
- 图表与数据分析页面成为主要迭代区域
- 需要显著提升视觉一致性

升级动作：

- 把 token 源拆分为多文件、按尺度归类
- 业务组件从"点状组件"扩展到"稳定模式库"
- 补充组件示例页和页面模式示例
- 增加更完整的工程检查规则

## 实施顺序

后续按以下顺序推进设计系统落地：

1. 先收口 token 源层与主题映射层
2. 再建立基础组件层（按需 add，不预先全量造）
3. 再抽业务组件
4. 再补工具链与可视化文档
5. 最后再做更强的工程约束与视觉回归
