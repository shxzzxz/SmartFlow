# 业务领域设计

## 1. 会计基础

SmartFlow 在数据层严格使用复式记账法，确保每一笔交易的资金流向都可追溯、可平账、可审计。

### 1.1 会计恒等式

所有数据流转遵循以下三个等式：

- **基础恒等式**：`资产 = 负债 + 所有者权益`
- **动态恒等式**：`资产 = 负债 + 所有者权益 + (收入 - 支出)`
- **记账操作等式**：`资产 + 支出 = 负债 + 所有者权益 + 收入`

记账操作等式将动态恒等式中的"-"消去，使所有变量都为非负数，是底层分录写入的形式基础。

### 1.2 五大账户类型

系统将所有"资金载体"和"收支分类"统一抽象为账户，分为五大类型：

| 账户类型 | 标识 | 业务含义 | 典型示例 |
| :--- | :--- | :--- | :--- |
| 资产 | `ASSET` | 资金的实际载体或应收款 | 现金、储蓄卡、第三方支付、基金、报销应收款 |
| 负债 | `LIABILITY` | 欠款或透支额度 | 信用卡、花呗、房贷、车贷、个人借款 |
| 所有者权益 | `EQUITY` | 净资产的调节项 | 期初余额、平账盈亏 |
| 收入 | `INCOME` | 资金流入的分类 | 工资、兼职、投资收益 |
| 支出 | `EXPENSE` | 资金流出的分类 | 餐饮、交通、房租 |

### 1.3 借贷映射规则

每一笔交易至少涉及两个账户，且**借方金额合计 = 贷方金额合计**。账户金额变化与借贷方向的映射关系如下：

| 账户类型 | 余额增加 | 余额减少 |
| :--- | :--- | :--- |
| 资产 / 支出 | 借（DEBIT） | 贷（CREDIT） |
| 负债 / 收入 / 所有者权益 | 贷（CREDIT） | 借（DEBIT） |

借贷方向是分录（`entries`）的核心字段，账户余额随分录过账自动累计。

## 2. 核心数据模型

本节列出表与关键字段；详细字段类型、索引与迁移在 drift schema 中定义。所有金额字段以最小货币单位（如人民币的"分"）作为整数存储。

### 2.1 accounts

承载五大类账户与收入 / 支出分类的单表树形结构。

```text
accounts
- id
- name
- account_type           ASSET / LIABILITY / EQUITY / INCOME / EXPENSE
- account_subtype        资产细分（CASH / BANK_CARD / THIRD_PARTY / INVESTMENT / REIMBURSEMENT）
                         负债细分（CREDIT_CARD / LOAN / CONSUMER_CREDIT）
                         其他类型为 NULL
- parent_id              树形父节点；顶级账户 / 顶级分类为 NULL
- currency_code          币种
- balance_minor          当前余额，由 entries 实时累计
- icon_key
- note
- credit_limit_minor     仅信贷账户使用
- billing_day            仅信贷账户使用
- repayment_day          仅信贷账户使用
- sort_order
- is_hidden              用户主动隐藏（仍可记账）
- archived_at            归档时间，非空表示该节点不再作为活跃写入目标
- system_key             系统语义身份，见下文
- created_at
- updated_at
```

**`system_key` 取值**：

- `OPENING_BALANCE` —— 期初余额对冲账户（EQUITY，按币种唯一）
- `REIMBURSEMENT_GAP_INCOME` —— 报销差额收入分类
- `IMPORT_FALLBACK` —— 导入兜底账户

约束：

- `system_key + currency_code` 唯一
- `system_key` 非空的节点不允许物理删除，可改名 / 改图标 / 改排序 / 隐藏

**层级与归档规则**：

- INCOME / EXPENSE 分类的活跃节点最多两级：一级分类、一级 / 二级分类
- 归档节点（`archived_at IS NOT NULL`）不能再作为父节点，也不能再作为新交易的写入目标
- 节点关系最大深度为 3：一级分类 → 二级子分类 → 归档节点
- 删除分类的两条路径：
  - 无任何 entries / 预算引用 → 物理删除
  - 存在历史引用 → 标记 `archived_at` 并将 `parent_id` 重指到迁移目标，历史 entries 不变
- 删除父分类前必须先处理所有活跃子分类，不允许递归级联迁移

### 2.2 transactions

业务事件主记录。

```text
transactions
- id
- business_purpose                  业务用途，见下文枚举
- occurred_at                       交易发生时间
- currency_code
- primary_amount_minor              交易头部展示金额；账务真相以 entries 为准
- counterparty_name                 交易对手方名称
- note                              用户备注
- parent_transaction_id             业务父链路（退款挂支出、报销到账挂报销垫付等）
- reimbursement_expense_account_id  仅报销垫付主链使用，记录原报销支出分类
                                    用于结束报销时少收差额复用同一分类
- mutation_kind                     ORIGINAL / CORRECTION / REVERSAL
- lifecycle_state                   ACTIVE / SUPERSEDED / VOIDED
- reverses_transaction_id           当前交易冲销哪一笔
- reversal_reason                   CORRECTION / DELETE
- corrects_transaction_id           当前更正交易替代哪一笔
- source_kind                       MANUAL / IMPORT / AUTO
- created_at
- updated_at
```

**`business_purpose` 枚举**：

```text
DAILY_EXPENSE             普通支出
DAILY_INCOME              普通收入
TRANSFER                  普通转账
REFUND                    退款
REIMBURSEMENT_ADVANCE     报销垫付
REIMBURSEMENT_RECEIPT     报销到账（部分到账）
REIMBURSEMENT_CLOSE       结束报销（最后一笔，可能含差额）
DEBT_REPAYMENT            还款
BORROWING                 借入
OPENING_BALANCE           期初余额
BALANCE_ADJUSTMENT        余额调整
```

冲销交易不单独使用枚举值，而是继承被冲销交易的 `business_purpose`，通过 `mutation_kind = REVERSAL` 区分。

**生命周期状态语义**：

- `ACTIVE` —— 当前有效记录
- `SUPERSEDED` —— 已被 CORRECTION 替代的旧记录
- `VOIDED` —— 已被 REVERSAL 冲销且未补记的记录（用户删除路径）

### 2.3 transaction_details

业务详情分项。承载收支统计的唯一真相。

```text
transaction_details
- id
- transaction_id
- line_no                       同一交易内排序
- label                         分项业务说明，受控枚举（见下）
- amount_minor                  分项金额
- stats_kind                    INCOME / EXPENSE / OTHER
- is_excluded_from_stats        默认 false。用户主动从收支统计中排除此分项
- is_excluded_from_budget       默认 false。用户主动从预算中排除此分项
- created_at
```

**`stats_kind` 语义**：

- `INCOME` —— 计入收入统计
- `EXPENSE` —— 计入支出统计
- `OTHER` —— 不计入收支统计（转账、还款本金、报销垫付主体、报销到账主体等内部资金流转）

**`label` 受控枚举**（按 `business_purpose` 分组）：

| business_purpose | 允许的 label |
| :--- | :--- |
| `DAILY_EXPENSE` | 主体支出 |
| `DAILY_INCOME` | 主体收入 |
| `TRANSFER` | 主体转账 / 转账手续费 |
| `REFUND` | 主体退款 |
| `REIMBURSEMENT_ADVANCE` | 垫付支出 |
| `REIMBURSEMENT_RECEIPT` | 单笔报销到账 |
| `REIMBURSEMENT_CLOSE` | 结束报销 / 报销差额支出 / 报销差额收入 |
| `DEBT_REPAYMENT` | 还款本金 / 还款利息 / 还款手续费 |
| `BORROWING` | 借入本金 |
| `OPENING_BALANCE` | 期初余额 |
| `BALANCE_ADJUSTMENT` | 余额调整 |

label 由系统按场景填充，不接受用户自由文本。

**与 entries 的关系**：

- `transaction_details` 与 `entries` **不互相外键引用**
- 二者都是 `transactions` 的不同视角阐释
- 同一笔交易，detail 数与 entry 数无对应关系

### 2.4 entries

复式记账分录。

```text
entries
- id
- transaction_id
- account_id
- direction              DEBIT / CREDIT
- amount_minor           允许负数，仅用于红字冲销
- created_at
```

约束：

- 同一 `transaction_id` 下，借方 `amount_minor` 合计 = 贷方 `amount_minor` 合计
- 红字冲销保留原借贷方向，`amount_minor` 取负数
- 写入后不允许 update / delete，更正与删除通过新交易冲销实现
- `account_id` 指向归档账户（archived）的 entries 仍合法存在，仅作为历史归集；新交易不得写入归档账户

### 2.5 budgets

月度预算。

```text
budgets
- id
- month_key             形如 202605，避免日期时区问题
- account_id            支出分类账户 id；NULL 表示当月总预算
- amount_minor
- currency_code
- created_at
- updated_at
```

约束：

- `account_id` 非空时只能指向 `account_type = EXPENSE` 的活跃账户
- `(month_key, account_id, currency_code)` 唯一；`account_id` 为 NULL 时按 `(month_key, currency_code)` 唯一
- 删除 / 迁移分类时，若该分类已有预算引用：预算迁移到目标分类；若目标分类同月已有预算，金额合并

## 3. 核心业务规则

### 3.1 一笔交易的完整表达

每一笔交易由三部分在同一数据库事务内提交：

- 1 条 `transactions` 主记录
- 1 条或多条 `transaction_details` 分项
- 2 条或多条 `entries` 分录

三者通过 `transaction_id` 关联，但 `transaction_details` 与 `entries` 不互相引用。

写入约束：

- entries 借贷必须平衡
- transaction_details 至少 1 条
- transaction_details 的 `amount_minor` 总和与 entries 的资金合计无强约束 —— 二者从不同视角阐释同一交易，可以不一致

### 3.2 更正机制

已写入的交易不直接更新原记录，更正分两步：

1. 生成一条 `mutation_kind = REVERSAL` 的冲销交易，`reverses_transaction_id` 指向原交易
2. 生成一条 `mutation_kind = CORRECTION` 的新交易，`corrects_transaction_id` 指向原交易，承载正确金额 / 账户 / 分类

原交易 `lifecycle_state` 设为 `SUPERSEDED`，默认列表不展示，详情页可展开历史链路。

### 3.3 红字冲销

冲销交易在 entries 上保留原借贷方向，`amount_minor` 取负数。

举例：原 `account=餐饮支出, direction=DEBIT, amount=100`，冲销分录写入 `account=餐饮支出, direction=DEBIT, amount=-100`。

效果：

- 账户余额抵销
- 借贷方向不翻转，便于审计和追溯
- transactions 头部金额仍记原始正值

### 3.4 删除机制

已写入交易不允许物理删除。"删除"统一走冲销路径：

- 生成一条 `mutation_kind = REVERSAL`、`reversal_reason = DELETE` 的冲销交易
- 原交易 `lifecycle_state` 设为 `VOIDED`
- 默认列表不展示原交易和冲销交易

`reversal_reason` 用于区分两种冲销动机：

- `CORRECTION` —— 用户在更正流程中产生的冲销
- `DELETE` —— 用户在删除流程中产生的冲销，且没有后续 CORRECTION

### 3.5 业务场景规则

本节描述每种业务场景的"应该如何表达"。具体表数据示例见《04. 业务场景流程示例》。

#### 3.5.1 普通支出

- `business_purpose = DAILY_EXPENSE`
- detail：1 条 `主体支出`，`stats_kind = EXPENSE`
- entries：支出分类账户 DEBIT，资金账户 CREDIT

#### 3.5.2 普通收入

- `business_purpose = DAILY_INCOME`
- detail：1 条 `主体收入`，`stats_kind = INCOME`
- entries：资金账户 DEBIT，收入分类账户 CREDIT

#### 3.5.3 普通转账

- `business_purpose = TRANSFER`
- detail：1 条 `主体转账`，`stats_kind = OTHER`；如有手续费再加 1 条 `转账手续费`，`stats_kind = EXPENSE`
- entries：转入资产账户 DEBIT，转出资产账户 CREDIT；如有手续费同时记一笔手续费支出分类 DEBIT 与资金账户 CREDIT

#### 3.5.4 退款

- `business_purpose = REFUND`
- `parent_transaction_id` 指向原支出交易
- detail：1 条 `主体退款`，`stats_kind = OTHER`
- entries：资金账户 DEBIT，原支出分类账户 CREDIT（红字效果由账户语义自然达成，不必使用红字写法）

#### 3.5.5 报销垫付

- `business_purpose = REIMBURSEMENT_ADVANCE`
- `reimbursement_expense_account_id` 记录用户为本次报销选择的支出分类
- detail：1 条 `垫付支出`，`stats_kind = OTHER`（垫付不计入个人支出）
- entries：报销应收账户（ASSET / REIMBURSEMENT 子类型）DEBIT，资金或负债账户 CREDIT

#### 3.5.6 报销到账（部分到账）

- `business_purpose = REIMBURSEMENT_RECEIPT`
- `parent_transaction_id` 指向 `REIMBURSEMENT_ADVANCE`
- detail：1 条 `单笔报销到账`，`stats_kind = OTHER`
- entries：资金账户 DEBIT，报销应收账户 CREDIT

#### 3.5.7 结束报销（含差额）

结束报销 = 剩余全额报销 + 差额收支。

- `business_purpose = REIMBURSEMENT_CLOSE`
- `parent_transaction_id` 指向 `REIMBURSEMENT_ADVANCE`
- detail 组合：
  - 1 条 `结束报销`，`stats_kind = OTHER`，金额为剩余应收金额
  - 若多收：再加 1 条 `报销差额收入`，`stats_kind = INCOME`
  - 若少收：再加 1 条 `报销差额支出`，`stats_kind = EXPENSE`
- entries：
  - 资金账户 DEBIT 实收金额
  - 报销应收账户 CREDIT 剩余应收金额
  - 多收差额：报销差额收入分类（懒加载 `system_key = REIMBURSEMENT_GAP_INCOME`）CREDIT 差额
  - 少收差额：原报销支出分类（取自 `transactions.reimbursement_expense_account_id`）DEBIT 差额

#### 3.5.8 还款（含本息）

- `business_purpose = DEBT_REPAYMENT`
- detail 组合：
  - 1 条 `还款本金`，`stats_kind = OTHER`
  - 若有利息：1 条 `还款利息`，`stats_kind = EXPENSE`
  - 若有手续费：1 条 `还款手续费`，`stats_kind = EXPENSE`
- entries：
  - 资金账户 CREDIT 总支付金额
  - 负债账户 DEBIT 本金
  - 利息支出分类 DEBIT 利息
  - 手续费支出分类 DEBIT 手续费

#### 3.5.9 借入

- `business_purpose = BORROWING`
- detail：1 条 `借入本金`，`stats_kind = OTHER`
- entries：资金账户 DEBIT，负债账户 CREDIT
- 债务期初导入（无对方资产）：资金侧由系统期初余额账户对冲

#### 3.5.10 期初余额

- `business_purpose = OPENING_BALANCE`
- detail：1 条 `期初余额`，`stats_kind = OTHER`
- entries：目标账户按其类型方向（资产 DEBIT / 负债 CREDIT）；对手方为系统期初余额权益账户（懒加载 `system_key = OPENING_BALANCE`）

#### 3.5.11 余额调整

- `business_purpose = BALANCE_ADJUSTMENT`
- detail：1 条 `余额调整`，`stats_kind = OTHER`
- entries：目标账户按差额方向调整，对手方为系统期初余额权益账户

### 3.6 统计规则

#### 3.6.1 账户维度

账户当前余额：从 entries 按 `account_id` 累计借贷净额，写入分录时同步更新 `accounts.balance_minor`。

账户流水：从 entries 反查，按 `transactions.occurred_at` 倒序，过滤 `lifecycle_state IN (SUPERSEDED, VOIDED)` 的 transactions。

#### 3.6.2 流水维度

默认流水列表：

- 来源 `transactions`，按 `occurred_at` 倒序
- 过滤条件：`lifecycle_state = ACTIVE` 且 `mutation_kind != REVERSAL`
- 子记录（退款、报销到账等）默认折叠到父交易详情中
- 详情页可展开冲销 / 更正历史链路

筛选维度：时间范围、账户、分类、business_purpose、关键字。

#### 3.6.3 收支统计维度

**收支统计的唯一真相在 `transaction_details`，不依赖 entries。**

月度总收入：

```text
SUM(amount_minor) FROM transaction_details
WHERE stats_kind = 'INCOME'
  AND is_excluded_from_stats = false
  AND transaction_id IN (本月 ACTIVE 且非 REVERSAL 的交易)
```

月度总支出：将 `stats_kind = 'INCOME'` 改为 `'EXPENSE'`。

#### 3.6.4 分类维度

按支出分类聚合 → 走 entries：

```text
SUM(entries.amount_minor) FROM entries
JOIN transactions ON entries.transaction_id = transactions.id
JOIN accounts ON entries.account_id = accounts.id
WHERE accounts.id IN (目标分类及其归档子节点)
  AND entries.direction = 'DEBIT'
  AND transactions.lifecycle_state = 'ACTIVE'
  AND transactions.mutation_kind != 'REVERSAL'
```

归档分类的历史 entries 自动归集到迁移目标，因为归档时 `parent_id` 已经重指。

按分类聚合时**不参与 detail 的 excluded 过滤** —— 排除标记仅作用于"收支统计"维度。如果未来需要"按分类的收支统计"同时遵守 excluded 过滤，可加 join `transaction_details`，但目前默认维度独立。

#### 3.6.5 预算维度

预算聚合走"分类维度"路径（entries），并叠加 `transaction_details.is_excluded_from_budget` 过滤：

```text
某分类某月已用预算 =
  SUM(entries.amount_minor)
  WHERE 命中分类树 AND direction = DEBIT
    AND transaction_id NOT IN (
      该月所有 detail 全部标 is_excluded_from_budget = true 的交易
    )
```

由于 entries 与 detail 不互相绑定，"细粒度排除"以**整笔交易粒度**应用：仅当一笔交易的所有 detail 都标记 `is_excluded_from_budget = true` 时，该交易的所有 entries 才从预算中排除。这与"用户在 transaction 维度勾选不计入预算 → 内部把所有 detail 同步勾选"的 UI 表达一致。
