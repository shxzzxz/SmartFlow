# 08.3 字段编辑三态协议

## 背景

partial update 命令对象常面临一个根本问题：可空字段的 `null` 值到底是"调用方没改这个字段"，还是"调用方明确要把这个字段清空"。两种语义不能用同一个 `null` 表达。

早期实现按"字段 + 配对的 `clearXxx: bool` 开关"区分：

```dart
class InstallmentContractPatch {
  final String? note;
  final InterestRatePeriod? interestRatePeriod;
  final int? interestRatePpm;
  final bool clearRate;             // true = 清空两个利率字段
  final bool clearNote;             // true = 清空 note
  final bool clearDisbursementAccount;
}
```

这种模式有几个真实问题：

1. **隐式约定容易出 bug**。例如曾出现过 `clearRate = (period == null && ppm == null)` 这样的推断——在"调用方每次传完整字段"的全量替换语义下成立，但一旦 command 变成 partial update（不传 = 不改），就会把"我只改了 note，没碰 rate"误判成"要清空利率"。
2. **配对 bool 数量随字段数线性增长**，每加一个允许清空的可空字段就要再加一个 `clearXxx`。
3. **协议在层与层之间容易割裂**。上层用 `noteChanged: bool` + `note: String?`，repository 又用 `clearNote: bool` + `note: String?`，service 不得不在边界做翻译；一旦翻译规则跑偏，整条链路上的"不改 / 设值 / 清除"就错位。

## 决策

跨层 partial update 协议统一采用 `Patch<T>` sealed class 表达三态：

```dart
sealed class Patch<T> {
  const factory Patch.set(T value) = PatchSet<T>;
  const factory Patch.clear() = PatchClear<T>;
}
```

字段类型为 `Patch<T>?`，三种取值语义：

| 取值 | 含义 |
| :--- | :--- |
| `null` | 不修改该字段 |
| `Patch.set(value)` | 把该字段设为 `value` |
| `Patch.clear()` | 把该字段置空 |

### 何时用 `Patch<T>?`、何时用 `T?`

- **业务确实允许"从有值清除为无值"的可空字段** → `Patch<T>?`。例：分期合同的 `note`、利率字段；`TransactionService.UpdateTransactionMetadataCommand.note`。
- **其他字段** → 用普通 `T?`，`null` 表示"不改"。例：分期合同的 `disbursementAccountId`（业务上禁止从有值清除——会跨合同 sourceType）；`firstRepaymentDate`（DB 列 non-null）；`isExcludedFromStats`（DB 列 non-null）。

判断准则：**列在 DB 中可为 NULL** **且** **业务上允许从有值改为 NULL** 同时成立才用 `Patch<T>?`；否则用 `T?`。不要给没有清除需求的字段加上 `Patch<T>?` 的负担。

### 协议层级一致

`Patch<T>?` 是跨层基础约定：service 命令对象、domain repository 接口、data 层 drift 仓库实现，使用同一份语义。**不允许在层与层之间做 `Patch<T>?` ↔ `noteChanged: bool` + `note: String?` 翻译**——一旦翻译，三态在层间被压回二态又展开，错位风险回来。

## 含义

- 命令对象（如 `UpdateTransactionMetadataCommand` / `UpdateContractCommand`）的可清除字段统一用 `Patch<T>?`。
- repository 接口同样用 `Patch<T>?`，drift 实现用 `switch` pattern 把 `Patch<T>?` 翻译成 drift 的 `Value<T?>`：

  ```dart
  Value<T?> _toNullableValue<T>(Patch<T>? patch) => switch (patch) {
    null => const Value.absent(),
    PatchSet<T>(:final value) => Value<T?>(value),
    PatchClear<T>() => Value<T?>(null),
  };
  ```

- 调用方在 nullable → 三态边界（如 `TransactionActionPolicy.changeNote(String? newNote)` 进入 service 时）展开为显式三元：

  ```dart
  note: newNote == null ? const Patch<String>.clear() : Patch.set(newNote)
  ```

  禁止用 "把传入的 nullable 一律映射成 set/clear" 的便捷工厂掩盖语义——`null` 在不同上下文含义不同（"不知道" / "不改" / "清除" / "占位"），把这个映射收成一个 helper 反而模糊语义、增加误用风险。

## 适用案例

- `lib/core/patch/patch.dart` — 协议定义。
- `lib/domain/services/transaction_service.dart` — `UpdateTransactionMetadataCommand.note`。
- `lib/domain/repositories/posting_repository.dart` — `updateTransactionMetadata.note`。
- `lib/domain/services/installment_service.dart` — `UpdateContractCommand.note / interestRatePeriod / interestRatePpm`；`EditRepaymentCommand.note`。
- `lib/domain/repositories/installment_repository.dart` — `InstallmentContractPatch.note / interestRatePeriod / interestRatePpm`；`InstallmentSchedulePatch.note`。
- `lib/data/repositories/drift_posting_repository.dart` 与 `drift_installment_repository.dart` — 把 `Patch<T>?` 翻译到 drift `Value<T?>`。

## 反例

### 反例一：每个允许清除的字段配一个 `clearXxx: bool`

- **形式**：`clearNote: bool` 与 `note: String?` 配对、`clearRate: bool` 与两个利率字段配对……
- **为什么不取**：约定隐式（两个字段必须一起读），扩展线性增长，且容易出 "`clearRate = (period == null && ppm == null)`" 那类把"不改"误判成"清除"的 bug。

### 反例二：在层与层之间翻译协议

- **形式**：service 命令用 `Patch<T>?`，repository 仍用 `clearXxx: bool` + `T?`；service 实现层做翻译胶水。
- **为什么不取**：三态信号在层间被压扁又还原，翻译规则一旦出错就错位；调用方查看 repository 接口与查看上游 command 接口看到的语义对不齐，理解负担大。

### 反例三：用 nullable → 三态的便捷工厂掩盖语义

- **形式**：`Patch.ofNullable(T?)` 之类的工厂——`null` 自动当成 `Patch.clear()`，非 null 自动当成 `Patch.set(...)`。
- **为什么不取**：`null` 在不同上下文含义不同。便捷工厂硬编码"`null` = clear"，调用方稍不留神就会把"不知道"或"不改"的 null 错填成"清除"。显式三元 `value == null ? Patch.clear() : Patch.set(value)` 让调用方必须明确选一种，更安全。

### 反例四：给所有可空字段都套 `Patch<T>?`

- **形式**：不论业务是否允许清除，把所有 nullable 字段都改成 `Patch<T>?`。
- **为什么不取**：调用方为没有"清除"需求的字段（如 `firstRepaymentDate: Patch<DateTime>?`）也得包 `Patch.set(...)`，增加噪音；同时把"业务允许清除"这个真实信号稀释了。`Patch<T>?` 是面向清除语义的工具，不是 nullable 的通用替代。

## 演进规则

- **新增需要清除语义的可空字段**：直接在命令对象 / patch 上声明 `Patch<T>?`，不要新加 bool 开关。
- **某字段从"普通 nullable"演化为"业务允许清除"**：把字段类型从 `T?` 改成 `Patch<T>?`，跟进所有调用点；不要保留半套旧协议。
- **跨编排层接入新 service**：所有 partial update 接口默认采用 `Patch<T>?` 协议，避免每个模块自造方言。
