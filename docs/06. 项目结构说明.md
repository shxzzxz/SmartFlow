# 项目结构说明

本文档定义 SmartFlow Flutter 工程的目录组织与分层边界。业务模型见《03. 业务领域设计》，视觉规范见《05. 设计系统》。

## 1. 总体原则

SmartFlow 采用集中领域层，而不是每个 feature 自带一套 `data/domain/usecases`。

复式记账规则会同时影响交易、账户、分类、预算、统计、报销和还款链路，因此核心规则集中在 `domain/`，持久化实现集中在 `data/`，feature 只负责页面与状态编排。

依赖方向：

```text
app / features / widgets
        ↓
domain
        ↓
data
```

约束：

- `domain/` 不依赖 Flutter、Riverpod、drift。
- `data/` 实现 `domain/repositories/` 中的接口。
- `features/` 通过 Riverpod provider 使用领域服务，不直接 new DAO / repository implementation。
- 不设置独立 `dependency_injection/` 目录；Riverpod provider 负责依赖装配。
- 主题、token、基础组件放 `design_system/`，不放进 `core/`。
- drift schema、DAO、迁移放 `data/database/`，不放进 `core/`。

## 2. 目录结构

```text
lib/
├── main.dart                         # 应用入口，启动 ProviderScope
├── app/
│   ├── app.dart                      # MaterialApp
│   ├── router.dart                   # go_router 路由
│   ├── providers.dart                # 顶层 Riverpod provider
│   └── bootstrap.dart                # 数据库打开、默认数据初始化等启动逻辑
├── core/                             # 无业务语义、无 UI 的跨层基础能力
│   ├── money/                        # Money 类型、按分计算、格式化
│   ├── time/                         # 日期、月份键、周期工具
│   ├── result/                       # Result / Failure
│   ├── errors/                       # 错误类型
│   └── id/                           # ID 生成策略
├── domain/                           # 领域层：复式记账规则与服务
│   ├── entities/                     # Account / Transaction / Entry / Budget 等领域实体
│   ├── enums/                        # 账户类型、业务用途、借贷方向、业务状态、分项类型等
│   ├── repositories/                 # 仓储接口
│   ├── services/                     # AccountService / TransactionService / BudgetService 等
│   ├── ledger/                       # 借贷映射、平账校验、统一过账
│   ├── stats/                        # 余额、收支、分类、预算统计
│   └── transaction_semantics/         # 业务用途到 details / entries 的构造规则
├── data/                             # 持久化层：drift 实现
│   ├── database/
│   │   ├── app_database.dart         # DriftDatabase
│   │   ├── tables/                   # accounts / transactions / entries / budgets 等表
│   │   ├── converters/               # 枚举、Money 等类型转换
│   │   └── migrations/               # 数据库迁移
│   ├── daos/                         # drift DAO
│   └── repositories/                 # domain repository 的 drift 实现
├── features/                         # 页面与状态编排，不放领域实体或仓储实现
│   ├── home/
│   ├── transactions/
│   ├── accounts/
│   ├── categories/
│   ├── budgets/
│   ├── stats/
│   ├── import/
│   ├── data_management/
│   └── settings/
├── design_system/                    # 项目级设计系统
│   ├── tokens/                       # 原始视觉 token
│   ├── theme/                        # ThemeData / ThemeExtension
│   ├── widgets/                      # 通用基础组件
│   └── showcase/                     # 示例页，可选
├── widgets/
│   └── business/                     # 跨 feature 复用的业务组件
└── l10n/                             # intl 资源
```

feature 内部按需使用：

```text
features/<feature>/
├── pages/                            # 页面级 widget
├── widgets/                          # 仅本 feature 使用的局部组件
├── providers/                        # Riverpod 状态与页面编排
└── view_models/                      # 表单、筛选、分页等 UI 模型
```

测试目录镜像核心分层：

```text
test/
├── core/
├── domain/
├── data/
└── features/
```

## 3. 目录规则

- `data/database/` 的表结构与字段命名对齐《03.1 数据模型》。
- 写交易必须在同一事务内落 `transactions + transaction_details + entries`，并同步更新账户余额。
- 复用到 2 个以上 feature 的业务组件迁移到 `widgets/business/`。
- `features/` 内不要新建 `data/`、`domain/`、`repositories/`。
- `domain/` 测试用纯 Dart unit test；`data/` 测试用 drift in-memory；`features/` 测试优先 mock provider。

## 4. 命名约定

- 文件与目录：`snake_case`。
- 类名：`UpperCamelCase`。
- 数据库列名：`snake_case`，与《03.1 数据模型》一致。
- 金额字段：数据库层用 `_minor` 后缀，领域 / UI 层使用 `Money`。
- Riverpod provider：`<scope>Provider`，如 `transactionListProvider`。
- 仅 feature 内使用的 provider 不导出到 feature 外。

## 5. 演进规则

- 新增字段：先改《03.1 数据模型》，再改 drift table、领域实体和 UI。
- 新增业务场景：先改《03.2 核心业务规则》和《03.3 业务场景流程示例》，再补 `domain/transaction_semantics/`。
- 新增 feature：优先只建 `pages/widgets/providers/view_models`。
- 性能优化：优先在 DAO / 查询层加索引、stream 或投影模型，不在页面层堆缓存。
