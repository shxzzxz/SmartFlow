# 项目结构说明

本文档定义 SmartFlow Flutter 工程的目录组织与分层边界。业务模型见《03. 业务领域设计》，视觉规范见《05. 设计系统》，业务域划分与依赖边界的决策依据见《08.1 业务域划分与账务核心独立性》。

## 1. 总体原则

SmartFlow 采用**集中领域层 + 业务域内部组织**。领域代码统一放 `lib/domain/`，但内部按业务域分子目录，而不是扁平铺开。

依赖方向：

```text
app / features / widgets
        ↓
domain/<域>
        ↓
domain/accounting   ← Shared Kernel（账务核心）
        ↓
data/<域>
```

约束：

- `domain/` 不依赖 Flutter、Riverpod、drift。
- `data/<域>/` 实现同名 `domain/<域>/repositories/` 中的接口。
- `features/` 通过 Riverpod provider 使用领域服务，不直接 new DAO / repository implementation。
- 不设置独立 `dependency_injection/` 目录；Riverpod provider 负责依赖装配。
- 主题、token、基础组件放 `design_system/`，不放进 `core/`。
- drift schema、DAO、迁移按业务域分目录放在 `data/<域>/`，统一注册到 `data/app_database.dart`。
- `features/` 内不要新建 `data/`、`domain/`、`repositories/`、`usecases/`。

## 2. 业务域与共享内核

`lib/domain/` 与 `lib/data/` 第一层目录是**业务域**。当前清单：

| 业务域 | 角色 | 说明 |
| :--- | :--- | :--- |
| `accounting`（账务核心） | **Shared Kernel** | 账户、交易、分录、过账规则、所有流图原语、查询投影 |
| `installments`（分期） | 独立业务域 | 分期合同、计划、还款 |
| `budgeting`（预算） | 独立业务域 | 预算（允许暂时薄，预留位置） |
| `analytics`（统计） | 独立业务域 | 统计能力，统计页消费此处 |

依赖与 import 边界详见《08.1 业务域划分与账务核心独立性》。关键规则：

- `accounting` 不 import 任何兄弟业务域；其它业务域单向依赖 `accounting`。
- 其它业务域只允许 import `accounting/{entities, enums, services}`，**不允许** import `accounting/{repositories, ledger}` 等内部目录。账务核心的查询能力必须通过 `services/` 暴露。
- 兄弟业务域之间不互相 import。
- domain/ 不反向依赖 features/ / app/ / data/（UI 适配、集成层不进 domain）。

## 4. 目录规则

### 跨业务域边界

- `domain/accounting/` 不能 import 任何兄弟业务域。
- 其它 `domain/<域>/` 只能 import `domain/accounting/{entities, enums, services}`，不能 import `domain/accounting/{repositories, ledger}` 等内部目录。
- 兄弟业务域之间不互相 import。
- `data/<域>/` 只 import 同业务域的 `domain/<域>/` 接口及 `domain/accounting/` 对外面（如需共享类型）。
- 这些规则应由 lint（custom_lint / dart_code_metrics）强制，违反即 CI 失败。

### 持久化

- 写交易必须在同一事务内落 `transactions + transaction_details + entries`，并同步更新账户余额。
- drift 表文件按业务域分目录放，但所有 `@DataClassName` / 表类统一注册到 `data/app_database.dart`（Drift 单库约束）。
- 迁移集中在 `data/migrations/`。

### features 与 widgets

- 复用到 2 个以上 feature 的业务组件迁移到 `widgets/business/`。
- `features/transactions/action_policy/` 集中放 TransactionActionPolicy 接口与各业务域实现；它是 UI 接入业务域的集成层，不属于任何业务域。

### 测试

- `domain/` 测试用纯 Dart unit test。
- `data/` 测试用 drift in-memory。
- `features/` 测试优先 mock provider。

## 5. 命名约定

- 文件与目录：`snake_case`。
- 类名：`UpperCamelCase`。
- 数据库列名：`snake_case`，与《03.1 数据模型》一致。
- 金额字段：数据库层用 `_minor` 后缀，领域 / UI 层使用 `Money`。
- Riverpod provider：`<scope>Provider`，如 `transactionListProvider`。
- 仅 feature 内使用的 provider 不导出到 feature 外。
- 业务域目录名用领域语言（`accounting / installments / budgeting / analytics`），不用技术词（`core / overlays / modules / engine`）。

## 6. 演进规则

### 新增字段
1. 先改《03.1 数据模型》；
2. 再改 `data/<域>/tables/` 的 drift table、`data/migrations/`；
3. 同步领域实体（`domain/<域>/entities/`）与 UI。

### 新增账务核心场景（流图原语）
1. 先改《03.2 核心业务规则》和《03.3 业务场景流程示例》；
2. 在 `domain/accounting/enums/` 扩展 `BusinessPurpose` / `TransactionDetailType` / `SystemKey`；
3. 在 `domain/accounting/services/` 与 `domain/accounting/ledger/` 落地过账规则。

### 新增独立业务域
按 08.1 的归类判断与演进规则：

1. 在 `domain/` 与 `data/` 新建 `<域>/` 子目录；
2. 在 08.1 的"业务域清单与角色"追加一行；
3. 如需接入通用 UI 动作槽位，在 `features/transactions/action_policy/` 新增 `<域>TransactionActionPolicy` 实现，定义 `owner_type / owner_role` 约定（参见 08.2）；
4. 在 `app/providers.dart` 注册 owner_type 到 policy 的映射。

### 新增 feature
- 优先只建 `pages/widgets/providers/view_models`；`action_policy/` 目前仅用于 transactions feature。
- 不要在 feature 内建 `domain/`、`data/`、`repositories/`、`usecases/`。

### 性能优化
- 优先在 DAO / 查询层加索引、stream 或投影模型，不在页面层堆缓存。
