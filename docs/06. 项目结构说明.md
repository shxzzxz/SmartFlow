# 项目结构说明

## 文档定位

本文档定义 SmartFlow Flutter 工程的目录组织、分层职责与命名约定，回答以下问题：

- 一段代码该写在哪一层
- 跨层之间通过什么接口对接
- 业务规则与持久化细节如何隔离
- 命名风格如何对齐

业务模型与领域规则见《03. 业务领域设计》。视觉规范与设计系统目录细节见《05. 设计系统》。

## 分层架构总览

延续原 Web 项目的四层骨架，按 Flutter 习惯收敛：

```
应用装配层（app/）
   ↓
表现层（features/<feature>/ + design_system/widgets/ + widgets/business/）
   ↓
领域层（domain/）          —— 业务规则、领域服务、仓储接口
   ↓
持久化层（data/）          —— drift schema、DAO、仓储实现
```

跨层规则：

- 上层依赖下层接口，禁止反向依赖
- `domain` 不引入 drift / Flutter / Riverpod 任何技术细节
- `data` 实现 `domain/repositories/` 中定义的接口
- `features/*/presentation` 通过 Riverpod provider 间接消费 domain 服务，不直接 new 仓储实现
- 横切能力（Money、错误类型、Result、日期工具等）放 `core/`，领域 / 表现 / 数据均可消费

## 顶层目录与职责

```
lib/
├── main.dart                       # 入口，仅做 ProviderScope 装配与运行
├── app/                            # 应用装配
├── core/                           # 跨层通用基础设施
├── data/                           # 持久化层（drift）
├── domain/                         # 领域层
├── features/                       # 业务特性（页面、状态、表单逻辑）
├── design_system/                  # 设计系统（token / theme / 基础组件）
├── widgets/
│   └── business/                   # 跨 feature 复用的业务组件
└── l10n/                           # 国际化资源
```

### app/

应用装配与全局接线。

```
app/
├── app.dart                        # MaterialApp / 根 Widget
├── router.dart                     # go_router 路由表
└── bootstrap.dart                  # 启动期初始化（数据库打开、默认数据初始化等）
```

职责：

- 组装 `MaterialApp`、注入 `ThemeData`（来自 `design_system/theme/`）
- 配置 go_router、绑定路由级守卫
- 初始化默认分类、系统语义账户（`OPENING_BALANCE` 等）
- 不承载业务规则；仅做"装配"

### core/

跨层通用基础。无业务语义、无 UI 依赖。

```
core/
├── money/                          # Money 类型（按分整数存储，封装四则运算与格式化）
├── result/                         # Result / Failure 类型
├── time/                           # DateTime 工具、月份键（month_key）转换
├── id/                             # ID 生成策略
└── errors/                         # 错误类型基类
```

约束：

- 不依赖 `domain` / `data` / `features`
- 不引入 drift、Riverpod、go_router 等具体框架（除非这是横切的本质用途）

### domain/

领域层。承载所有"如何记账"的规则，且对持久化方案无感。

```
domain/
├── entities/                       # 领域实体
│   ├── account.dart                # accounts 行为模型
│   ├── transaction.dart
│   ├── transaction_detail.dart
│   ├── entry.dart
│   └── budget.dart
├── enums/
│   ├── account_type.dart           # ASSET / LIABILITY / EQUITY / INCOME / EXPENSE
│   ├── account_subtype.dart
│   ├── business_purpose.dart       # DAILY_EXPENSE / TRANSFER / REFUND / ...
│   ├── mutation_kind.dart          # ORIGINAL / CORRECTION / REVERSAL
│   ├── lifecycle_state.dart
│   ├── direction.dart              # DEBIT / CREDIT
│   ├── stats_kind.dart             # INCOME / EXPENSE / OTHER
│   └── system_key.dart             # OPENING_BALANCE / REIMBURSEMENT_GAP_INCOME / IMPORT_FALLBACK
├── repositories/                   # 仓储接口（由 data 层实现）
│   └── ledger_repository.dart
├── services/                       # 领域服务
│   ├── account_service.dart
│   ├── category_service.dart
│   ├── transaction_service.dart
│   ├── transaction_query_service.dart
│   ├── budget_service.dart
│   ├── import_service.dart
│   └── default_category_initializer.dart
├── ledger/
│   ├── ledger_rules.dart           # 借贷映射规则、平账校验
│   └── posting.dart                # 统一过账流程：写 transaction + details + entries
├── stats/
│   ├── income_expense_stats.dart   # 收支统计：走 transaction_details
│   └── balance_stats.dart          # 余额、分类、预算统计：走 entries
└── transaction_semantics/
    ├── purpose_label_rules.dart    # business_purpose → 允许的 detail label 映射
    └── scenario_builders/          # 各业务场景的 detail / entries 构造（见 03 文档 3.5）
```

关键服务（对齐《03. 业务领域设计》）：

- **AccountService**：账户树管理，归档（写 `archived_at`、重指 `parent_id`）、`system_key` 唯一约束
- **CategoryService**：收入 / 支出分类管理（本质是 `accounts` 子树）
- **TransactionService**：业务输入 → `posting.dart` 统一过账，写入 transaction + details + entries
- **TransactionQueryService**：组装流水读模型，过滤 `lifecycle_state`、折叠子记录
- **BudgetService**：预算 CRUD、分类迁移时的预算合并
- **LedgerRepository**：领域对持久化的唯一访问入口

约束：

- 服务实现以纯 Dart 编写，可单独 unit test
- 不引入 `package:drift`、`package:flutter`、`package:riverpod`
- 数据库事务能力通过仓储接口暴露（如 `runInTransaction(action)`），由 data 层的 drift 实现

### data/

持久化层。drift 是当前唯一实现。

```
data/
├── database/
│   ├── app_database.dart           # drift @DriftDatabase，组装表 + DAO + 迁移
│   ├── tables/
│   │   ├── accounts_table.dart
│   │   ├── transactions_table.dart
│   │   ├── transaction_details_table.dart
│   │   ├── entries_table.dart
│   │   └── budgets_table.dart
│   ├── converters/                 # JSON / 枚举 ↔ 列类型
│   └── migrations/                 # MigrationStrategy 与版本步骤
├── daos/
│   ├── accounts_dao.dart
│   ├── transactions_dao.dart
│   ├── transaction_details_dao.dart
│   ├── entries_dao.dart
│   └── budgets_dao.dart
└── repositories/
    └── drift_ledger_repository.dart  # 实现 domain/repositories/ledger_repository.dart
```

约束：

- 表结构与字段命名严格对齐《03. 业务领域设计》（snake_case + 整数分 + 枚举字符串）
- 仓储实现负责 drift 实体 ↔ 领域实体的转换；领域层只见领域实体
- 任何写入路径必须在事务内同时落 transaction + details + entries
- 写入 entries 后同步累加 `accounts.balance_minor`，由仓储或 DAO 层完成

### features/

按业务特性组织 UI 与状态。每个 feature 自治，跨 feature 复用走 `widgets/business/` 与 `domain/`。

```
features/
├── home/
│   ├── presentation/               # 页面 widget（HomePage 等）
│   ├── providers/                  # Riverpod state
│   └── view_models/                # 表单状态、列表筛选状态等（必要时）
├── transactions/
│   ├── presentation/               # TransactionListPage / TransactionEditorPage / ...
│   ├── providers/
│   └── view_models/
│       ├── transaction_form_state.dart
│       ├── transaction_filter_state.dart
│       └── transaction_pagination.dart
├── accounts/
│   ├── presentation/
│   └── providers/
├── categories/
│   ├── presentation/
│   └── providers/
├── budgets/
│   ├── presentation/
│   └── providers/
├── stats/
│   ├── presentation/               # 收支统计、趋势图、同环比
│   └── providers/
├── import/
│   ├── presentation/               # 第三方账单导入（一木记账等）
│   ├── providers/
│   └── parsers/                    # 解析器实现（对照 docs/analysis_rules/）
├── data_management/                # 数据导出 / 清理
│   ├── presentation/
│   └── providers/
└── settings/
    ├── presentation/
    └── providers/
```

每个 `presentation/` 内部按需自分子目录（`widgets/`、`sheets/`、`dialogs/`），不强制统一。

约束：

- feature 内部的 widget 仅服务于本 feature
- 一旦同类表达在 2 个以上 feature 复用，迁移到 `widgets/business/`
- providers 只承载 UI 编排状态，业务规则下沉到 `domain/services/`

### design_system/

按《05. 设计系统》定义。

```
design_system/
├── tokens/                         # 原始视觉值
├── theme/                          # ColorScheme / ThemeExtension / ThemeData
├── widgets/                        # 通用基础组件（AppButton / AppCard / ...）
└── showcase/                       # 示例页（可选，供 review 对照）
```

### widgets/business/

跨 feature 复用的业务组件层。命名直接体现财务语义。

```
widgets/business/
├── money_text.dart                 # 金额展示
├── money_input.dart                # 金额输入
├── account_type_tag.dart
├── transaction_purpose_badge.dart
├── transaction_lifecycle_badge.dart
└── business_icon.dart
```

约束（详见《05. 设计系统》第 4 层）：

- 业务组件命名必须体现财务语义
- 业务组件可以消费 ThemeExtension，但不允许定义新 token
- 不允许业务组件直接持有 Riverpod state；状态由 feature 层注入

### l10n/

intl + flutter_localizations 资源。

```
l10n/
├── intl_en.arb
├── intl_zh.arb
└── ...
```

## 测试目录

测试镜像 `lib/` 目录结构。

```
test/
├── core/
├── domain/
│   ├── ledger/
│   ├── services/
│   └── transaction_semantics/
├── data/
│   ├── repositories/               # 仓储集成测试（drift in-memory）
│   └── migrations/                 # 迁移路径测试
└── features/
    └── <feature>/                  # widget test / provider test
```

约束：

- `domain/` 测试用纯 Dart unit test，不挂载 drift
- `data/` 测试以 drift in-memory 实例运行，覆盖事务、唯一约束、迁移
- `features/` widget test 优先 mock provider，不落库

## 与原 Web 项目的对应关系

| 原项目（Vue 3 + Dexie） | 本项目（Flutter + drift） |
| :--- | :--- |
| `src/app/services.ts` | `lib/app/bootstrap.dart` + Riverpod 顶层 provider |
| `src/views/` | `lib/features/*/presentation/` |
| `src/components/business/` | `lib/widgets/business/` |
| `src/components/ui/` | `lib/design_system/widgets/` |
| `src/composables/use*.ts` | `lib/features/*/providers/` |
| `src/domain/services/` | `lib/domain/services/` |
| `src/domain/repositories/` | `lib/domain/repositories/` |
| `src/domain/ledger.ts / posting.ts` | `lib/domain/ledger/` |
| `src/db/` | `lib/data/database/` |
| `src/db/repositories/dexie-ledger-repository.ts` | `lib/data/repositories/drift_ledger_repository.dart` |
| `src/lib/` | `lib/core/` + `lib/domain/transaction_semantics/` |
| `src/types/` | `lib/data/database/tables/`（drift 推导）+ `lib/domain/entities/` |

迁移到 Flutter 后做了以下结构调整：

- `composables` 拆为 feature 内的 `providers/` 与 `view_models/`，不再共享一个全局目录
- 原 `src/lib/` 中纯工具（如 `utils.ts`）下沉到 `core/`；含领域语义的（如 `transaction-semantics.ts`、`ledger-format.ts`）归入 `domain/`
- 表现层按 feature 切分，避免原项目 `views/` + `composables/` + `components/` 三处分散

## 命名约定

### 文件与目录

- 文件 / 目录名：snake_case（Dart 标准）
- 类名：UpperCamelCase
- 数据库列名：snake_case，与《03. 业务领域设计》逐字一致

### 类型与枚举

- 领域实体：`Account` / `Transaction` / `TransactionDetail` / `Entry` / `Budget`
- 枚举值：屏内大写（`AccountType.asset`），落库时映射为大写字符串（`ASSET`），由 drift TypeConverter 完成
- 仓储接口：`<Aggregate>Repository`，实现：`Drift<Aggregate>Repository`
- 服务：`<Domain>Service`

### Riverpod provider

- 命名：`<scope>Provider`，如 `accountListProvider`、`transactionFormProvider`
- 仅在 feature 内可见的 provider 不导出到 feature 外
- 跨 feature 复用的 provider 上移到 `app/` 或 `domain/` 同级的 `providers/` 目录（按需新增）

### Money

- 字段名：以 `_minor` 结尾标识"按分存整数"（与表字段一致）
- 类型：领域 / UI 层用 `Money`；DB 层用 `int`，由仓储边界转换

## 演进策略

- 新增 feature：先创建 `features/<name>/presentation/` 与 `providers/`，业务规则补到 `domain/services/`
- 新增字段：先改《03. 业务领域设计》→ 再改 drift table → 再改领域实体 → 最后接表现层
- 新增业务场景：补 `domain/transaction_semantics/scenario_builders/<purpose>.dart`，并在《04. 业务场景流程示例》登记
- 抽业务组件：同一表达 2 处以上重复时，从 feature 内迁移到 `widgets/business/`
- 性能瓶颈：先在 DAO 加索引或 stream 化，避免在表现层做缓存
