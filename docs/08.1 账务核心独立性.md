# 08.1 账务核心独立性

## 适用范围

本决策**只**约束"业务编排叠加层"，不约束"账务流图原语"。两者的定义见下文。如果对一项新功能的归类有歧义，按本文末尾的"归类判断"做。

## 背景

SmartFlow 账务核心（`transactions / transaction_details / entries` + `transaction_service / posting_service` + `BusinessPurpose / TransactionDetailType / SystemKey` 等枚举）承担过账、平账、更正、删除、统计口径等基础职责。

在账务核心之上扩展业务有两种形态：

### A. 账务流图原语（Accounting Flow Primitives）

这类功能**引入了新的 entry / details 拆分模式**——它们的"业务含义"与"过账规则"绑定，离开账务核心无法表达。

已识别的流图原语，及其与核心枚举的耦合点：

| 原语 | `BusinessPurpose` | `TransactionDetailType` | 备注 |
| :--- | :--- | :--- | :--- |
| 转账 | `transfer` | `transferMain` / `transferFee` | 转出 / 转入 + 可选手续费 |
| 退款 | `refund` | `refundMain` | 反向冲减原支出 |
| 报销垫付 / 收款 / 结案 | `reimbursementAdvance` / `reimbursementReceipt` / `reimbursementClose` | `reimbursementAdvanceMain` / `reimbursementReceiptMain` / `reimbursementCloseMain` / `reimbursementGapExpense` / `reimbursementGapIncome` | 三段式 + 缺口拆分；额外依赖 `SystemKey.reimbursementGapIncome` |
| 借入 / 还款 | `borrowing` / `debtRepayment` | `borrowingPrincipal` / `repaymentPrincipal` / `repaymentInterest` / `repaymentFee` / `repaymentDiscount` | 本金 + 利息 + 手续费 + 优惠四元拆分 |
| 余额初始化 / 调整 | `openingBalance` / `balanceAdjustment` | `openingBalanceMain` / `balanceAdjustmentMain` | 与 `SystemKey.openingBalance` 配合 |

流图原语**理应**进核心枚举与服务实现，它们是会计模型的一部分。本决策不要求剥离任何一个。

### B. 业务编排叠加层（Business Orchestration Overlay）

这类功能**不引入**新的 entry 模式，只是在已有原语之上挂载独立的状态机 / 计划表 / 关联实体，把"哪些交易归我管、按什么节奏发生、什么时候算结束"组织起来。

典型形态（用通用术语描述，与具体业务解耦）：

- 业务上有一个**主体实体**（合同 / 订阅 / 目标 / 凭证…），描述一段长期协议或承诺；
- 主体可能附带**计划表**（多期未来支出 / 收入 / 还款的 expectation）；
- 实际发生时，**关联表**把一笔账务交易绑定到主体上；
- 主体有自己的**状态机**（active / settled / closed / expired …）。

落地候选包括但不限于：分期合同、信用卡账单分期、订阅扣款、保险缴费、定投计划、储蓄目标、租金合同。

当编排叠加层需要"撤销/修改受管交易"的能力时，会出现两种诱惑：

1. **枚举侵入**——在 `BusinessPurpose` 中追加 `<module>Specific` 枚举，让账务核心"识别"该模块；
2. **核心级联**——在 `transactionService.deleteTransaction` / `correctTransaction` 中检测交易是否被某模块持有，命中则反向调用模块的 service 做收尾。

两条路都会把"该模块的存在"写进账务核心，造成跨模块耦合，且让"新增一种业务"变成"改一次核心层"。

## 决策

**业务编排叠加层不污染账务核心。**

具体表现为：

- 编排叠加层的 `service / repository / 枚举 / 表` 不进入 `transaction_service.dart`、`posting_service.dart` 等账务核心文件，也不进入 `BusinessPurpose / TransactionDetailType / SystemKey` 等核心枚举。
- 账务核心**不**`import` 任何编排叠加层模块。
- `BusinessPurpose` 不为编排叠加层做"模块身份"特化（不存在为模块识别而生的枚举）。
- 编排叠加层负责自身（主体 / 计划 / 关联表 / 状态机）与账务核心（交易表）之间的一致性。

### 调用方向

```text
feature/UI ─→ 编排叠加层 service
                   │
                   ↓
            账务核心 service (transaction / posting)
```

- 编排叠加层**单向依赖**账务核心，不反向。
- 账务核心**不知道**编排叠加层的存在。

具体的数据层契约、服务层 API、修改路径收敛手段、测试边界等落地方案，在每个编排叠加层接入时按场景单独讨论，不在本决策中预设。

## 反例

以下方案违反本决策，遇到类似诱惑时**不**采用。

### 反例一：把编排叠加层身份编码进 `BusinessPurpose`

- **形式**：在核心枚举中追加 `<module>Specific` 项（如 `installmentBorrowing` / `subscriptionCharge` / `insurancePremium`），仅为让调用方"知道这笔交易归 X 模块"。
- **为什么不取**：
  - 账务过账规则没有任何依赖该区分的逻辑（叠加层的交易与"非叠加层版本"过账拆分完全一致），等于为模块识别加了一个错位的钩子；
  - 核心枚举变成编排叠加层的注册表，每加一个模块都要改核心；
  - 历史数据迁移代价高，且未来废弃模块时枚举无法回收。
- **与流图原语的区别**：`reimbursementAdvance` 等枚举之所以合理在核心，是因为它们改变了 detail 拆分模式（缺口拆分、三段式过账规则）；编排叠加层不改 detail 拆分。

### 反例二：账务核心做"业务感知级联"

- **形式**：在 `transactionService.deleteTransaction` / `correctTransaction` 中检查叠加层是否持有该交易，命中则反向调用叠加层 service 做收尾。
- **为什么不取**：
  - 账务核心反向依赖编排叠加层，无法独立编译 / 测试；
  - 每加一个叠加层都要改核心 API 实现，违背开闭原则；
  - "删交易"是核心原语，被业务编排盖住后语义不再纯粹（同一个 API 在不同上下文做不同事）。

### 反例三：在通用入口绕过约束做"智能联动"

- **形式**：检测到交易归叠加层管，仍然让通用层直接编辑 / 删除，背后再隐式调用叠加层 service 同步状态。
- **为什么不取**：跨模块协同逻辑藏在调用方的隐式回调里，难以审计；调用方对动作影响范围没有清晰预期；编排规则更新时同步逻辑容易漂移。

## 归类判断

新功能接入前，按以下三问判断它属于 A（流图原语）还是 B（编排叠加层）：

1. 它是否引入了现有 entry / details 拆分之外的新模式？
   - 是 → 流图原语，**应当**进核心枚举与 `transaction_service`，并补充对应 `detail_type` / `SystemKey`。
   - 否 → 继续问 2。
2. 它是否有独立于交易的状态机 / 关联实体（合同、订阅、目标、计划等）？
   - 是 → 编排叠加层，独立 service / repository / 数据表，遵守本决策。
   - 否 → 继续问 3。
3. 把它从代码中删掉，账务核心能否独立完成过账？
   - 不能 → 流图原语。
   - 能 → 编排叠加层。

任何让账务核心 `import` 编排叠加层模块的方案 → 拒绝。任何让 `BusinessPurpose` 增长以服务编排叠加层身份识别的方案 → 拒绝。

## 演进规则

- 新增编排叠加层时，先在本文件追加"适用案例"指向其业务设计文档，再就具体接入方案（数据表、反查、修改入口收敛等）单独讨论。
- 现有流图原语（transfer / refund / reimbursement / borrowing / opening 等）**不**剥离；它们的核心耦合是合理的。
- 若未来某个流图原语演化出独立状态机（例如报销引入审批流），可在其上**追加**一个编排叠加层，但原有 detail 拆分仍留在核心。
