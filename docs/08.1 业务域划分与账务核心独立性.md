# 08.1 业务域划分与账务核心独立性

## 适用范围

本决策约束 SmartFlow 的业务域划分与边界：

- 哪些功能应归入**账务核心**（账务流图原语），哪些应作为**独立业务域**；
- 业务域如何在 `lib/domain/` 与 `lib/data/` 中组织；
- 业务域之间的依赖方向与 import 边界。

不约束：

- 任一业务域内部的 entities / services / repositories 细分（保留战术 DDD 自由度）；
- `app/`、`core/`、`design_system/`、`widgets/`、`features/` 的组织（按各自规则）；
- UI 与业务域的接入协议（见《08.2 通用 UI 与业务域接入协议》）。

## 背景

SmartFlow 账务核心（`transactions / transaction_details / entries` + `transaction_service / posting_service` + `BusinessPurpose / TransactionDetailType / SystemKey` 等枚举）承担过账、平账、更正、删除、统计口径等基础职责。

在账务核心之上扩展业务时，会出现两类形态——

### A. 账务流图原语（Accounting Flow Primitives）

这类功能**引入了新的 entry / details 拆分模式**——它们的"业务含义"与"过账规则"绑定，离开账务核心无法表达。

已识别的流图原语，及其与核心枚举的耦合点：

| 原语 | `BusinessPurpose` | `TransactionDetailType` | 备注 |
| :--- | :--- | :--- | :--- |
| 转账 | `transfer` | `transferMain` / `transferFee` | 转出 / 转入 + 可选手续费 |
| 退款 | `refund` | `refundMain` | 反向冲减原支出 |
| 报销垫付 / 收款 / 结案 | `reimbursementAdvance` / `reimbursementReceipt` / `reimbursementClose` | `reimbursementAdvanceMain` / `reimbursementReceiptMain` / `reimbursementCloseMain` / `reimbursementGapExpense` / `reimbursementGapIncome` | 三段式 + 缺口拆分；额外依赖 `SystemKey.reimbursementGapIncome` |
| 借入 / 还款 | `borrowing` / `debtRepayment` | `borrowingPrincipal` / `repaymentPrincipal` / `repaymentInterest` / `repaymentFee` / `repaymentDiscount` | 本金 + 利息 + 手续费 + 优惠四元拆分 |
| 余额初始化 / 调整 | `openingBalance` / `balanceAdjustment` | `openingBalanceMain` / `balanceAdjustmentMain` | 与 `SystemKey.openingBalance` 配合 |

流图原语**理应**进核心枚举与服务实现，它们是会计模型的一部分。本决策不要求剥离任何一个。

### B. 独立业务域（Independent Business Domain）

这类功能**不引入**新的 entry 模式，只是在已有原语之上挂载独立的状态机 / 计划表 / 关联实体，把"哪些交易归我管、按什么节奏发生、什么时候算结束"组织起来。

典型形态（用通用术语描述，与具体业务解耦）：

- 业务上有一个**主体实体**（合同 / 订阅 / 目标 / 凭证…），描述一段长期协议或承诺；
- 主体可能附带**计划表**（多期未来支出 / 收入 / 还款的 expectation）；
- 实际发生时，**关联表**把一笔账务交易绑定到主体上；
- 主体有自己的**状态机**（active / settled / closed / expired …）。

落地候选包括但不限于：分期合同、信用卡账单分期、订阅扣款、保险缴费、定投计划、储蓄目标、租金合同。

当独立业务域需要"撤销/修改受管交易"的能力时，会出现两种诱惑：

1. **枚举侵入**——在 `BusinessPurpose` 中追加 `<module>Specific` 枚举，让账务核心"识别"该业务域；
2. **核心级联**——在 `transactionService.deleteTransaction` / `correctTransaction` 中检测交易是否被某业务域持有，命中则反向调用该业务域的 service 做收尾。

两条路都会把"该业务域的存在"写进账务核心，造成跨域耦合，且让"新增一种业务"变成"改一次核心层"。

### 与 DDD 术语的对应

本决策采用 DDD 的战略设计框架：

| 本文用语 | DDD 标准术语 |
| :--- | :--- |
| 业务域（总称） | Bounded Context |
| 账务核心（也称账务域） | Shared Kernel |
| 其它业务域（分期 / 预算 / 统计 / ...） | 单向依赖共享内核的 Bounded Context |
| 业务域之间的依赖规则 | Context Map（Customer-Supplier） |

行文优先使用中文术语；附注 DDD 原词只为术语锚定。

## 决策

### 1. 业务域分类与边界

**账务核心独立。其它业务域单向依赖账务核心，账务核心不感知其它业务域的存在。**

具体表现为：

- 其它业务域的 `service / repository / 枚举 / 表` 不进入 `transaction_service.dart`、`posting_service.dart` 等账务核心文件，也不进入 `BusinessPurpose / TransactionDetailType / SystemKey` 等核心枚举。
- 账务核心**不**`import` 任何其它业务域的模块。
- `BusinessPurpose` 不为其它业务域做"模块身份"特化（不存在为业务域识别而生的枚举）。
- `transactions.owner_type / owner_id / owner_role` 是开放字符串形式的业务归属索引，只保存"这笔交易归哪个业务域 / 主体 / 角色"，不进入过账规则，也不让账务核心解释具体含义。
- 其它业务域负责自身（主体 / 计划 / 关联表 / 状态机）与账务核心（交易表）之间的一致性。

### 2. 业务域清单与角色

`lib/domain/` 与 `lib/data/` 的第一层目录就是业务域。当前清单：

| 业务域 | 角色 | 说明 |
| :--- | :--- | :--- |
| `accounting`（账务核心） | **Shared Kernel** | 账户、交易、分录、过账规则、所有流图原语、查询投影。其它业务域消费它，它不知道其它业务域存在 |
| `installments`（分期） | 独立业务域 | 分期合同、计划、还款 |
| `budgeting`（预算） | 独立业务域 | 预算实体与服务（允许暂时薄，预留位置） |
| `analytics`（统计） | 独立业务域 | 趋势 / 洞察 / 预测等统计能力，统计页消费此处 |

加新业务域按"演进规则"流程，并在本表追加一行。

### 3. 调用方向与 import 边界

**调用方向**：

```text
features / UI ─→ 其它业务域 service
                       │
                       ↓
                 账务核心 service
```

- 其它业务域**单向依赖**账务核心，不反向。
- 账务核心**不知道**其它业务域的存在。
- 兄弟业务域之间**不互相依赖**（假设当前一笔交易只有一个 owner）。

**显式 import 规则**（应由 lint / custom_lint 强制，违反即 CI 失败）：

| 目录 | 是否允许被其它业务域 import |
| :--- | :--- |
| `domain/accounting/entities/*` | ✓（类型契约） |
| `domain/accounting/enums/*` | ✓（契约） |
| `domain/accounting/services/*` | ✓（唯一对外能力入口） |
| `domain/accounting/repositories/*` | ✗（账务核心内部抽象，外部需要查询时应让 services 加方法） |
| `domain/accounting/ledger/*` | ✗（内部规则） |

> `TransactionActionPolicy` 不在此表中——它是 UI 接入业务域的集成层契约，整套位于 `features/transactions/action_policy/`，不属于任何业务域。详见 08.2。

兄弟业务域之间不互相 import；若未来出现真实交叉需求，通过共享 ID + 账务核心串联，不直接 import。

### 4. 账务引擎心智模型

`domain/accounting/` 是"账务引擎"的当前形态。"引擎"指它对外只暴露一组明确的能力（services + 类型契约），内部规则（ledger、posting 细节、内部数据结构）对外不可见。

不为此新增 `engine/` 子目录——`accounting/services/` 即引擎对外面。"账务引擎"是命名约定与心智模型，不是物理结构。

**长期方向**：当 `accounting/` 稳定且其它业务域足够多，可整体提炼为独立 pub 包（`packages/smartflow_accounting_engine/`）。届时迁移成本主要是 pubspec 与少量 import 路径，目录形态不变。**当前不做此提炼**。

### 5. Drift 单库约束

Drift 要求所有表注册到同一个 `@DriftDatabase`。本决策的应对：

- 表文件按业务域分目录放（`data/<域>/tables/`），但 `app_database.dart` 统一 import 并注册；
- 迁移仍集中在 `data/migrations/`（schema 物理上是一个）；
- 物理 schema 统一，逻辑归属按业务域区分；
- 跨域边界靠 import 规则维持，不靠物理隔离。

未来若真需要把某业务域物理隔离（独立 sqlite 文件 / 独立 db），是另一项决策，不在本范围。

## 归类判断

新功能接入前，按以下三问判断它属于 A（账务流图原语）还是 B（独立业务域）：

1. 它是否引入了现有 entry / details 拆分之外的新模式？
   - 是 → 流图原语，**应当**进核心枚举与 `transaction_service`，并补充对应 `detail_type` / `SystemKey`。
   - 否 → 继续问 2。
2. 它是否有独立于交易的状态机 / 关联实体（合同、订阅、目标、计划等）？
   - 是 → 独立业务域，独立 service / repository / 数据表 / 目录，遵守本决策。
   - 否 → 继续问 3。
3. 把它从代码中删掉，账务核心能否独立完成过账？
   - 不能 → 流图原语。
   - 能 → 独立业务域。

任何让账务核心 `import` 其它业务域模块的方案 → 拒绝。任何让 `BusinessPurpose` 增长以服务业务域身份识别的方案 → 拒绝。任何让 `owner_*` 参与核心过账分支的方案 → 拒绝。任何把账务核心的 `repositories / ledger` 等内部目录暴露给其它业务域的方案 → 拒绝。

## 反例

以下方案违反本决策，遇到类似诱惑时**不**采用。

### 反例一：把业务域身份编码进 `BusinessPurpose`

- **形式**：在核心枚举中追加 `<module>Specific` 项（如 `installmentBorrowing` / `subscriptionCharge` / `insurancePremium`），仅为让调用方"知道这笔交易归 X 业务域"。
- **为什么不取**：
  - 账务过账规则没有任何依赖该区分的逻辑（业务域的交易与"非业务域版本"过账拆分完全一致），等于为业务域识别加了一个错位的钩子；
  - 核心枚举变成业务域的注册表，每加一个业务域都要改核心；
  - 历史数据迁移代价高，且未来废弃业务域时枚举无法回收。
- **替代方案**：使用 `owner_type / owner_id / owner_role` 表达业务归属；`BusinessPurpose` 继续只表达账务行为。
- **与流图原语的区别**：`reimbursementAdvance` 等枚举之所以合理在核心，是因为它们改变了 detail 拆分模式（缺口拆分、三段式过账规则）；独立业务域不改 detail 拆分。

### 反例二：账务核心做"业务感知级联"

- **形式**：在 `transactionService.deleteTransaction` / `correctTransaction` 中检查某业务域是否持有该交易，命中则反向调用业务域 service 做收尾。
- **为什么不取**：
  - 账务核心反向依赖业务域，无法独立编译 / 测试；
  - 每加一个业务域都要改核心 API 实现，违背开闭原则；
  - "删交易"是核心原语，被业务感知盖住后语义不再纯粹（同一个 API 在不同上下文做不同事）。

### 反例三：在通用入口绕过约束做"智能联动"

- **形式**：检测到交易归某业务域管，仍然让通用层直接编辑 / 删除，背后再隐式调用业务域 service 同步状态。
- **为什么不取**：跨域协同逻辑藏在调用方的隐式回调里，难以审计；调用方对动作影响范围没有清晰预期；业务规则更新时同步逻辑容易漂移。
- **替代方案**：通用 UI 通过 `TransactionActionPolicy` 路由到业务域 service（见 08.2）。

### 反例四：回到扁平 `domain/services/`、`domain/entities/`

- **形式**：把所有 service 平铺在 `domain/services/`，靠文件名前缀（`installment_*` / `budget_*`）区分业务域。
- **为什么不取**：
  - "账务核心独立"无法用目录表达，违反就靠 review 抓；
  - 加新业务域会让顶层目录文件膨胀到几十个，再分组的迁移成本更高；
  - "集中"被误读为"扁平"，结构隐式鼓励耦合。

### 反例五：每 feature 自带 `domain/` 与 `data/`

- **形式**：在 `features/installments/` 下重建 `domain/`、`data/`、`usecases/`，把分期所有代码收进 feature。
- **为什么不取**：
  - `features/` 是 UI 与状态编排层，承担领域代码会让 feature 不再是 UI 单元；
  - 跨 feature 共享的领域规则（账务核心）没有合适位置；
  - 这是 doc 06 早期就拒绝的反模式，本决策不回头。

### 反例六：为账务核心起特殊目录名（`core / kernel / shared / engine`）

- **形式**：把账务核心放 `domain/core/` 或 `domain/kernel/`，与 `domain/installments/` 形成"一等公民 + 二等公民"的目录关系。
- **为什么不取**：
  - 核心的特殊性来自**依赖方向**（被所有人依赖、不依赖任何人），不是目录名；
  - 起特殊名意味着加新业务域时要决策"它算核心吗"，引入分类负担；
  - "engine"、"core" 等词与 `lib/core/`（无业务工具层）混淆。
- **替代**：用领域词 `accounting`，靠 lint 规则维持其"被依赖、不依赖"的实际地位。

### 反例七：按"流图原语 vs 业务域"在 `accounting/` 内部再分目录

- **形式**：`domain/accounting/primitives/transfer/`、`domain/accounting/primitives/reimbursement/`，把账务核心按业务用途拆得很细。
- **为什么不取**：
  - 流图原语之间共享大量过账规则与实体，强行拆分会重复或交叉 import；
  - 战术 DDD 的 `entities / services / ledger` 分组已经够用；
  - 核心的复杂度是"规则的复杂度"，不是"上下文的复杂度"，不应套用业务域级别的拆分。

### 反例八：在 `accounting/` 下建 `engine/` 子目录

- **形式**：`domain/accounting/engine/transaction_service.dart`。
- **为什么不取**：
  - "engine" 是心智模型，不是物理结构——`accounting/services/` 本身就是引擎对外面；
  - 加一层 `engine/` 会让"哪些算引擎、哪些不算"成为新的分类问题；
  - 真要把引擎独立成 pub 包时，整体迁移 `accounting/` 比挑出 `engine/` 子集更清晰。

## 演进规则

### 加新业务域的流程

1. 创建 `domain/<域>/` 与 `data/<域>/` 目录；
2. 在本决策"业务域清单与角色"追加一行；
3. 如需接入通用 UI 的动作槽位，在 `features/transactions/action_policy/` 新增对应业务域的 `XxxTransactionActionPolicy` 实现，定义 `owner_type / owner_role` 约定（参见 08.2）；
4. 在 `app/providers.dart` 注册 owner_type 到 policy 的映射；
5. lint 规则按"调用方向与 import 边界"自动覆盖新业务域。

### 现有流图原语不剥离

`transfer / refund / reimbursement / borrowing / opening` 等**不**剥离；它们的核心耦合是合理的，归 `domain/accounting/`。

### 流图原语演化出独立状态机时

例如报销引入审批流。可在其上**追加**一个独立业务域（新建 `domain/approval/` 等），但原有 detail 拆分仍留在核心。

### 何时把业务域提炼为 pub 包

- 业务域代码量超过约 30 个文件且对外 API 稳定数月；
- 出现明确复用诉求（另一个 app / package 需要该业务域）；
- 团队规模扩大、需要并行开发隔离。

未达到这些条件之前，单仓多目录结构足够。

### 不回归扁平结构

集中扁平的成本随业务域数量超线性增长；分业务域组织的成本是固定的（一次性）。
