# 分期借贷业务设计

本文承接《03. 业务领域设计》，针对**分期借贷**场景给出概念模型、数据模型、业务规则与场景示例。

SmartFlow 定位为记账类应用，分期能力以**记录与追踪**为目标，不承担金融计算职责。

## 1. 业务背景与定位

### 1.1 目标能力

- 在负债账户下管理"分期合同"，覆盖**贷款分期**（借呗类）与**账单分期**（花呗类）两种来源
- 从负债账户视角可查看挂载的分期合同列表
- 从合同视角可查看预期还款计划
- 还款提示基于计划表的预期还款日
- 支持**提前还部分本金**（影响后续期次的本金 / 利息分配）与**提前结清**

### 1.2 不做的事

- **不**承诺与各家产品完全一致的还款金额。系统按标准公式根据用户输入的分期方式 / 期数 / 利率生成**初始计划**，但费率换算、计息口径、节假日顺延等产品差异由用户**手工编辑计划行**兜底。
- **不**保留计划表的修改痕迹。计划本身就是"预期"，可随时被提前还款 / 调整费率覆盖。
- **不**承担逾期罚息的自动计算，由用户在实际还款时手工补录手续费。

### 1.3 与现有能力的关系

- 沿用 `accounts` 中的 `LIABILITY` 账户类型，无需新增账户子类型；贷款类账户使用 `LOAN`，信用类账户使用 `CONSUMER_CREDIT` / `CREDIT_CARD`
- 沿用 `BORROWING`、`DEBT_REPAYMENT` 两个 `business_purpose`，无需新增枚举
- 沿用 `REPAYMENT_PRINCIPAL` / `REPAYMENT_INTEREST` / `REPAYMENT_FEE` / `REPAYMENT_DISCOUNT` 分项类型
- 分期合同是叠加在交易模型之上的**独立业务域**，不影响 entries 借贷平衡或统计口径

## 2. 概念模型

### 2.1 两类合同来源

| 来源类型 | 标识 | 典型场景 | 是否生成放款交易 |
| :--- | :--- | :--- | :--- |
| 放款类 | `DISBURSEMENT` | 借呗、网商贷、银行消费贷 | 是，资金账户与负债账户各记一笔 |
| 账单转换类 | `BILL_CONVERSION` | 花呗账单分期、信用卡账单分期 | 否，本金来自已存在的消费负债 |

**核心差异**：放款类在合同创建瞬间有真实的资金 / 负债变动；账单转换类只是改变了已有负债的还款节奏，账面无变动。

### 2.2 三层结构

```text
installment_contracts        合同：基本协议信息、状态
        │
        ├── installment_schedules    计划：N 期预期还款条目，可编辑
        │
        └── installment_repayments   实际：每期实际还款，关联 transaction
```

### 2.3 与交易表的关系

- 合同创建（放款类）：写入 1 笔 `BORROWING` transaction，并由合同记录其 id
- 合同创建（账单转换类）：**不写交易**
- 每期实际还款：写入 1 笔 `DEBT_REPAYMENT` transaction，并由 `installment_repayments` 记录关联
- 实际还款金额（本金 / 利息 / 手续费）**只**通过关联 transaction 的 `transaction_details` 表达，`installment_repayments` 不冗余存金额

## 3. 数据模型

### 3.1 installment_contracts

分期合同主表。

```text
installment_contracts
- id
- liability_account_id           挂载的负债账户 id（LIABILITY）
- source_type                    DISBURSEMENT / BILL_CONVERSION
- disbursement_account_id        放款入账资产账户 id；仅 DISBURSEMENT 非空
- disbursement_transaction_id    放款交易 id；仅 DISBURSEMENT 非空
- principal_minor                合同本金
- total_periods                  总期数
- start_date                     合同起始日期
- repayment_method               还款方式枚举
- interest_rate_period           利率周期：ANNUAL / MONTHLY / DAILY；用户可空
- interest_rate_ppm              利率数值，单位 ppm（1 ppm = 0.0001%）；用户可空
- status                         ACTIVE / SETTLED / CLOSED
- note
- created_at
- updated_at
```

**`repayment_method` 取值**：

- `EQUAL_INSTALLMENT` —— 等额本息
- `EQUAL_PRINCIPAL` —— 等额本金
- `INTEREST_FIRST` —— 先息后本
- `FLAT_FEE` —— 一次性手续费（常见于花呗账单分期）
- `CUSTOM` —— 自定义（完全由用户编辑计划行驱动）

`repayment_method` 仅作为**计划生成时的初始化模板**和展示标签，生成后不再约束计划内容；用户编辑计划行不会回写 `repayment_method`。

**`interest_rate_period` + `interest_rate_ppm` 说明**：

- 两字段同填或同空，用户可按产品惯例选择利率口径（借呗常给日利率、银行贷款常给年利率）
- 单位 ppm（百万分之一）= 0.0001%；如年利率 7.2% = 72000 ppm，日利率 0.025% = 250 ppm
- 仅在合同创建生成初始计划时参与计算，以及在合同详情页展示；生成后即使用户编辑计划行也不回写本字段，**视作"用户当初的输入快照"**，不作为账务真相

**`status` 取值**：

- `ACTIVE` —— 进行中
- `SETTLED` —— 全部期次按计划还清
- `CLOSED` —— 提前结清或合同终止

约束：

- `liability_account_id` 指向的账户必须 `account_type = LIABILITY` 且未归档
- `source_type = DISBURSEMENT` 时，`disbursement_account_id` 与 `disbursement_transaction_id` 必须非空
- `source_type = BILL_CONVERSION` 时，`disbursement_account_id` 与 `disbursement_transaction_id` 必须为空
- 合同删除：仅当无任何 `installment_repayments` 关联且无放款 transaction 时允许物理删除；否则必须走"提前结清"路径

### 3.2 installment_schedules

预期还款计划表。

```text
installment_schedules
- id
- contract_id
- period_no                      期序号，从 1 起
- expected_repayment_date        预期还款日
- expected_principal_minor       预期本金
- expected_interest_minor        预期利息
- expected_fee_minor             预期手续费
- status                         PENDING / PAID / SKIPPED
- note
- created_at
- updated_at
```

**`status` 取值**：

- `PENDING` —— 待还
- `PAID` —— 已关联实际还款，期次完成
- `SKIPPED` —— 因提前结清等原因不再需要还款

约束：

- `(contract_id, period_no)` 唯一
- 计划行允许直接编辑金额 / 日期 / 状态，不保留修改痕迹
- 计划行不参与账户余额或统计口径

### 3.3 installment_repayments

实际还款关联表。

```text
installment_repayments
- id
- contract_id
- repayment_type                 REGULAR / EXTRA_PRINCIPAL / EARLY_SETTLEMENT
- schedule_id                    对应 installment_schedules.id；仅 REGULAR 非空
- transaction_id                 对应的 DEBT_REPAYMENT transaction id
- created_at
- updated_at
```

**`repayment_type` 取值**：

- `REGULAR` —— 计划内的某期还款，必须绑定 `schedule_id`
- `EXTRA_PRINCIPAL` —— 计划外提前还本金，`schedule_id` 为空；触发后续 PENDING 期次重算
- `EARLY_SETTLEMENT` —— 提前结清，`schedule_id` 为空；触发剩余 PENDING 批量 SKIPPED 并将合同置为 CLOSED

约束：

- `repayment_type = REGULAR` 时 `schedule_id` 非空，且 `(contract_id, schedule_id)` 唯一 —— 一期只关联一笔实际还款交易
- `repayment_type = EXTRA_PRINCIPAL / EARLY_SETTLEMENT` 时 `schedule_id` 必须为空
- `transaction_id` 指向的 transaction 必须 `business_purpose = DEBT_REPAYMENT`
- 不冗余存金额；金额查询通过关联 transaction 的 `transaction_details` 获取

## 4. 业务规则

### 4.1 合同创建

**放款类（`source_type = DISBURSEMENT`）**：

用户输入：放款入账资产账户、贷款账户、本金、起始日、分期方式（`repayment_method`）、期数、利率（`interest_rate_period` + `interest_rate_ppm`）。FLAT_FEE 方式下利率可空，由用户在生成的计划行中调整手续费。

1. 在同一数据库事务内：
   - 写入 1 笔 `BORROWING` transaction（沿用《03.2》1.5.9 规则）
   - 写入 1 条 `installment_contracts`，`disbursement_transaction_id` 指向上一步交易
   - 根据 `repayment_method`、`total_periods`、`start_date`、`principal_minor`、`interest_rate_period` + `interest_rate_ppm` 批量生成 `installment_schedules`
2. 计划生成是**一次性模板填充**，生成后系统不再根据利率重算；用户可对任意行编辑

**账单转换类（`source_type = BILL_CONVERSION`）**：

用户输入：信用账户、本金（即被转换的账单金额）、起始日、分期方式、期数、利率或每期手续费。

1. 在同一数据库事务内：
   - **不写交易**
   - 写入 1 条 `installment_contracts`，`disbursement_account_id` 与 `disbursement_transaction_id` 为空
   - 批量生成 `installment_schedules`
2. 转换前的原始消费交易**不做关联标记**——分期视角只关心还款进度，原始消费明细仍在交易流水中可查

### 4.2 计划编辑

- 用户可直接编辑 `installment_schedules` 行的金额、日期、`note`
- 编辑不写历史
- 仅 `status = PENDING` 的行允许金额 / 日期编辑；`status = PAID` 的行只能通过更正其关联还款交易来调整

### 4.3 期次还款

每期实际还款（`repayment_type = REGULAR`）：

1. 在同一数据库事务内：
   - 写入 1 笔 `DEBT_REPAYMENT` transaction（沿用《03.2》1.5.8 规则），其 `transaction_details` 拆分本金 / 利息 / 手续费 / 优惠
   - 写入 1 条 `installment_repayments`，`repayment_type = REGULAR`，`schedule_id` 指向对应计划行，`transaction_id` 指向上一步交易
   - 将该 `installment_schedules.status` 置为 `PAID`
2. 检查合同状态：若该合同所有 `installment_schedules` 均为 `PAID` / `SKIPPED`，将 `installment_contracts.status` 置为 `SETTLED`

> 当期实际利息 / 手续费 / 优惠与计划行金额的差异由用户在还款表单中直接填入，不回写计划行，也不视为"提前还款"。

### 4.4 提前还款

提前还款分两种形态：

**提前还部分本金（`repayment_type = EXTRA_PRINCIPAL`）**：

- 写一笔 `DEBT_REPAYMENT`，主要构成为 `REPAYMENT_PRINCIPAL`，可附带 `REPAYMENT_FEE`（如部分提前还款手续费）
- 写 `installment_repayments`，`repayment_type = EXTRA_PRINCIPAL`，`schedule_id = NULL`
- **重算后续 PENDING 期次**：
  - 剩余本金 = `principal_minor` − 已 `PAID` 期次的 `expected_principal_minor` 累计 − 历次 `EXTRA_PRINCIPAL` 的 `REPAYMENT_PRINCIPAL` 累计
  - 剩余期数 = `total_periods` − 已 `PAID` 期数
  - 按合同 `repayment_method` + 当前利率重算所有 `status = PENDING` 的计划行（金额覆盖，日期保留）
  - `PAID` / `SKIPPED` 行不动
  - 用户先前对 PENDING 行的手工编辑会被覆盖；UI 应在执行前展示"重算预览"
- 合同状态保持 `ACTIVE`

**提前结清（`repayment_type = EARLY_SETTLEMENT`）**：

- 写一笔 `DEBT_REPAYMENT` 表达本次结清金额，其中可包含违约金（计入 `REPAYMENT_FEE`）
- 写 `installment_repayments`，`repayment_type = EARLY_SETTLEMENT`，`schedule_id = NULL`
- 将所有未还期次 `installment_schedules.status` 置为 `SKIPPED`
- 合同状态置为 `CLOSED`

违约金 / 提前还款手续费**只**进入还款交易的 `REPAYMENT_FEE` 分项，不回写计划表。

### 4.5 还款交易的更正与删除

- 更正某期还款：沿用《03.2》1.2 更正机制写入 REVERSAL + CORRECTION；`installment_repayments.transaction_id` 同步重指到 CORRECTION 交易
- 删除某期还款：沿用《03.2》1.4 删除机制写入 REVERSAL；通过 `transaction_id` 反查 `installment_repayments`，再按 `repayment_type` 联动：
  - `REGULAR` —— 通过 `schedule_id` 直接定位计划行，置回 `PENDING`，删除 repayments 行
  - `EXTRA_PRINCIPAL` —— 删除 repayments 行后**重新触发 PENDING 期次重算**（撤销了提前还本金的效果）
  - `EARLY_SETTLEMENT` —— 删除 repayments 行，把因结清被置 `SKIPPED` 的期次恢复为 `PENDING`，合同状态从 `CLOSED` 回退至 `ACTIVE`
- 合同状态联动：还款被撤销后，若合同曾置为 `SETTLED`，回退至 `ACTIVE`

### 4.6 合同删除与归档

- 物理删除仅允许在合同**无任何关联还款 transaction**且**放款交易也尚未写入**（账单转换类天然满足后者）的情况
- 已发生还款的合同必须走"提前结清"路径，最终落到 `CLOSED` 状态，保留历史

**Step 2 — 重算后续 PENDING 期次**：
