# 08.2 通用 UI 与编排叠加层接入协议

## 适用范围

本决策约束**通用 UI**（如交易详情页、列表页）与**编排叠加层**（08.1 定义；分期、订阅、保险等）之间的接入方式。

不约束：
- UI 与账务核心 / 流图原语的关系（按 `BusinessPurpose` switch 渲染照常）；
- 编排层各自内部的 UI（如分期合同详情页内的渲染逻辑）。

本决策落地的代码骨架（`TransactionHandlers` / `OrchestrationContribution` / `OrchestrationRegistry`）截至 v0.1 文档版尚未实施，决策意图先沉淀，实施按节奏安排。

## 背景

08.1 已定 UI → 编排叠加层 → 账务核心 的单向依赖关系，但**没有规定** UI **如何** 接入编排层——具体到"通用 UI 怎么知道某笔交易归哪个编排层管"、"该走什么删除 / 编辑路径"等。

当前实践（分期模块，截至本文档发布时）：
- `transaction_detail_page` 调用 `installmentService.findLinkByTransaction` 判定"这笔归不归分期管"；
- 命中则把整个 `ActionBar` 替换为单一跳转按钮（"去分期合同详情"）；
- 业务字段（日期、账户）的编辑用 `if (linked) blocked-toast`，引导用户去合同页修改；
- 元数据（备注）仍走通用入口。

这套方案能跑，但有两个问题：

1. **加新编排层成本高**：所有"是否被持有"的判定中心化在通用 UI 一个文件里。第二个编排层出现时，要新增 link provider + 新增 ActionBar 变体 + 给每个字段 onTap 加新分支。每个编排层在通用 UI 里"刻一份名字"。
2. **用户操作被打断**：受管交易的"删除 / 编辑"被替换为"去 X 页"。从用户视角，"分期还款"和"普通还款"的行为差异远大于必要——本可以在原页面就地完成的操作（如撤销还款、修改账户）被推到另一处。

> 注：UI 直接 `import` 编排层 service **不违反** 08.1（08.1 只禁止账务核心 import 编排层）。本决策的动机是 **演进成本** 与 **用户体验**，不是 08.1 的延伸禁令。

## 决策

通用 UI 通过 **三个抽象** 与编排叠加层解耦：

### 抽象一：`TransactionHandlers` —— universal 动作的"槽集"

定义所有交易都拥有的固定形状方法。当前清单：

- `delete()` —— 删除当前交易
- `editRoutePath()` —— 编辑入口的路由路径
- `changeSettlementAccount(accountId)` —— 改 paidFrom / receive 账户
- `changeOccurredAt(time)` —— 改交易时间
- `changeNote(note)` —— 改备注
- `canEdit(field)` —— 字段级可用性（返回带 reason 的 `EditPermission`）

### 抽象二：`OrchestrationContribution` —— 编排层贡献装配

每个编排层实现一个 `OrchestrationContribution`，给定 `transactionId`：
- 若该笔归本编排层管 → 返回装配好的 `TransactionHandlers`；
- 否 → 返回 `null`。

### 抽象三：`OrchestrationRegistry` —— 装配聚合

启动时收集所有 `OrchestrationContribution`。UI 查 registry：
- 首个命中即返回该 handler；
- 全部不命中 → UI 使用 `DefaultTransactionHandlers`（走 `transactionService`）。

### 动作的二分

| 类型 | 决定可见性 | 决定具体行为 | 例子 |
| :--- | :--- | :--- | :--- |
| **purpose-bound** | `BusinessPurpose`（UI 按 switch 渲染） | 流图原语 service | 退款（dailyExpense / reimbursementAdvance）、报销结案（reimbursementAdvance） |
| **universal** | 永远可见 | handler 提供 | 删除、编辑跳转、改账户、改时间、改备注 |

**purpose-bound 动作不进 `TransactionHandlers`**。它们由通用 UI 按 `BusinessPurpose` 自行渲染，跟编排层无关——编排叠加层持有的交易类型自然不会触发与之无关的 purpose switch（如分期持有的 `borrowing` / `debtRepayment` 不会触发退款渲染）。

### 调用方向

```text
通用 UI ─→ TransactionHandlers (interface)
              ▲
              │  provided by
              │
   ┌──────────┴───────────────┐
   │                          │
DefaultHandlers       OrchestrationContribution ─→ 编排层 service
(走 transactionService)
```

通用 UI **只 import** `TransactionHandlers` / `OrchestrationRegistry`，不 import 任何具体编排层模块。每个 `XxxContribution` 在 Provider 装配时注册到 registry。

### 委托模式

编排层 handler **不要求** 重写所有 universal 动作。它只重写"会影响编排状态"的动作：
- 分期还款的 `delete` → 必须走 `revertRepayment`（自动回退 schedule 状态）
- 分期还款的 `editRoutePath` → 必须指向分期专属的编辑页

其他动作（如改 paidFrom 账户、改时间、改备注）若不破坏编排状态，handler **直接委托给 `transactionService`**：

```dart
@override
Future<Result<void>> changeNote(String? note) =>
    _fallback.updateTransactionMetadata(...);  // 不重写，直接借道
```

Handler 只承担"差异"，不重复"共性"。

## 反例

### 反例一：通用 UI 维护 `if (link != null) replace-ActionBar` 分支

- **形式**：`transaction_detail_page` 持有 `installmentLink` / `subscriptionLink` 等多个状态，按 link 类型嵌套 if/else 替换 UI 组件。
- **为什么不取**：
  - 每加一个编排层都要改通用 UI，违背开闭原则；
  - 通用 UI 被迫识别每个编排层的身份，认知边界随编排层数量线性膨胀；
  - 跨编排层的 UI 一致性靠各分支自觉，容易漂移。

### 反例二：把 purpose-bound 动作塞进 `TransactionHandlers`

- **形式**：handler 接口里加 `refund()` / `closeReimbursement()` 之类的方法，每个编排层都给 no-op。
- **为什么不取**：
  - 混淆 universal 与 purpose-bound 的归属；
  - 编排层 handler 被迫实现一堆"不该自己管"的方法；
  - 接口随 purpose 数量膨胀；
  - 当某编排层的持有范围扩大时（假设未来出现"分期支出"），仍可通过演进规则的"可选 slot" 优雅扩展，不需要现在预设。

### 反例三：用 redirect 替代就地操作

- **形式**：受管交易的 ActionBar 只放"去 X 页"的跳转按钮，用户被踢出当前页面才能完成删除 / 编辑等操作。
- **为什么不取**：
  - 同类交易在不同上下文行为不一致，认知打断；
  - 用户无法在原位完成原本就该原位完成的操作；
  - "跳转"本身可以作为 handler 提供的可选动作（如"在合同详情中查看"），与"撤销""修改"等并列，**但不应是唯一动作**。

### 反例四：让通用 UI 装配 handler

- **形式**：通用 UI 自己写 `if (link is InstallmentRepaymentLink) handlers = InstallmentHandlers(...); else if (...) ...`。
- **为什么不取**：
  - 通用 UI 又开始识别编排层身份，退化回反例一；
  - 装配逻辑（"这笔归我管 + 该如何处理"）属于编排层自身，不属于 UI；
  - 通用 UI 的单元测试被迫 stub 每个编排层。

## 归类判断

新动作 / 新功能接入 UI 时，按以下次第判断归属：

1. **它是否依赖某个具体 `BusinessPurpose`？**
   - 是 → **purpose-bound**，进通用 UI 的 purpose switch，跟 handler 无关。
2. **它是否所有交易都有 / 都可能有？**
   - 是 → **universal**，进 `TransactionHandlers` 接口；编排层按需重写，默认实现走 `transactionService`。
3. **它是否只属于某个编排层的"主体"操作？**
   - 是 → 不进通用 UI，归编排层自己的 feature 页面（如合同详情页）。

任何让通用 UI `import` 具体编排层模块 / 识别具体 link 子类型的方案 → 拒绝。  
任何让 `TransactionHandlers` 接口为 purpose-bound 动作开**不可选**槽位的方案 → 拒绝。

## 演进规则

- **加新 universal 动作**：扩 `TransactionHandlers` 接口 + 所有现有实现跟进。
- **加新编排层**：新建一个 `XxxContribution`，注册到 registry 一行；通用 UI 一行不动。
- **某 purpose-bound 动作未来需要被编排层覆盖**（如假设出现"分期支出"，dailyExpense 的退款要走分期版本）：`TransactionHandlers` 增加一个 **可选 slot**（默认 null = 走原 purpose-bound 渲染），handler 选择性重写。
- **跨编排层冲突**（一笔交易理论上同时被多个编排层持有）：当前 registry 按"首个命中即返回"，假设互斥；若未来真有交叉需求，再扩展为"动作合并"模式，不预先设计。
- **handler 实例化时机**：按需（进入交易详情页时），不预 cache 全列表。Contribution 的反查 `findLinkByTransaction` 是异步轻量查询。
